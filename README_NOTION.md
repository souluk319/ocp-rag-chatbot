# OCP 운영/교육 가이드 RAG 챗봇

노션에 바로 붙여넣어 읽기 쉽게 정리한 버전이다.  
오늘 작업 내용을 복기하면서 "이 프로젝트가 왜 이렇게 설계되었는가"를 빠르게 이해하는 데 목적이 있다.

---

## 1. 이 프로젝트를 한 문장으로 설명하면

폐쇄망 환경에서 OpenShift 문서를 기반으로,  
운영 질문과 교육 질문에 대해 **한국어로**, **근거를 붙여**, **follow-up까지 처리하는** RAG 챗봇을 만드는 프로젝트다.

---

## 2. 이 프로젝트가 어려운 이유

겉으로 보면 그냥 "문서 넣고 챗봇 만들기"처럼 보이지만 실제 난이도는 다르다.

이 프로젝트는 동시에 아래 문제를 풀어야 한다.

1. 문서를 잘 가져와야 한다.
2. 문서를 검색하기 좋은 형태로 정리해야 한다.
3. 질문을 검색 질의로 잘 바꿔야 한다.
4. BM25와 벡터 검색을 잘 합쳐야 한다.
5. 검색된 문서 중 진짜 근거만 답변에 써야 한다.
6. follow-up에서 "그거", "그 복구" 같은 표현을 해석해야 한다.
7. 애매하면 억지로 답하지 않고 clarification 해야 한다.
8. citation은 내부 문서 링크로 연결되어야 한다.

즉 이건 단순한 챗봇이 아니라  
`문서 처리 + 검색 시스템 + 답변 정책 + 대화 UX`를 같이 다루는 프로젝트다.

---

## 3. 왜 Part 1 ~ Part 4로 쪼갰는가

이 프로젝트는 4단계로 나뉜다.

### Part 1. 전처리

역할:
- 문서 수집
- HTML 정리
- 섹션 추출
- 청킹
- 임베딩
- Qdrant 적재

핵심 질문:
- 문서를 어떤 단위로 잘라야 나중에 잘 찾히는가?

### Part 2. 검색

역할:
- 질의 정규화
- follow-up rewrite
- BM25 검색
- vector 검색
- hybrid fusion

핵심 질문:
- 사용자의 질문에 맞는 책과 청크를 제대로 찾는가?

### Part 3. 답변

역할:
- 검색된 문서 중 일부만 context로 조립
- LLM 호출
- citation 정리
- ambiguous/no-answer 처리

핵심 질문:
- 찾은 근거만으로 안전하게 대답하는가?

### Part 4. UI

역할:
- 실제 질문을 넣어보는 QA 콘솔
- follow-up 확인
- citation 클릭
- regenerate / reset

핵심 질문:
- 점수 말고 실제 사용자가 보기엔 괜찮은가?

### 왜 이렇게 나눴냐

검색이 망했는데 답변 프롬프트를 고치고 있으면 삽질이다.  
답변 정책이 문제인데 retrieval만 손보면 또 삽질이다.

그래서 실패 원인을 분리하려고 일부러 4단계로 쪼갰다.

---

## 4. 현재 구조를 가장 짧게 요약하면

```text
html-single 문서
-> 정규화
-> 헤더 기반 청킹
-> 임베딩 + BM25 코퍼스 생성
-> BM25 검색 + vector 검색
-> RRF 기반 hybrid fusion
-> context gate
-> LLM answer + citation
-> QA UI
```

---

## 5. Part 1에서 제일 중요한 설계 포인트

### 5-1. 왜 html-single를 쓰는가

문서 하나를 통째로 가져오기 좋고,  
heading / anchor / 본문 구조를 비교적 온전히 보존할 수 있기 때문이다.

### 5-2. 왜 헤더 기반 청킹인가

사용자의 질문은 보통 문서 전체가 아니라 특정 섹션과 대응된다.

예를 들어:
- `etcd 백업은 어떻게 해?`
- `OpenShift 아키텍처를 설명해줘`

이 질문들은 각각 문서의 특정 section을 찾는 문제다.  
그래서 먼저 heading 단위로 자르고, 너무 큰 섹션만 토큰 기준으로 2차 분할하는 전략을 쓴다.

### 5-3. 현재 청킹 전략

1. 문서를 section 단위로 나눈다.
2. 각 section 안에서 문단 블록으로 묶는다.
3. `chunk_size`를 넘는 블록만 추가 분할한다.
4. `[CODE]`, `[TABLE]` 블록은 최대한 안 깨고 유지한다.

현재 기본값:
- `chunk_size = 160`
- `chunk_overlap = 32`

### 5-4. 왜 이 값으로 줄였나

이전에는 더 큰 청크를 썼는데,  
긴 블록이 tokenizer에 너무 크게 들어가면서 길이 초과 경고가 났다.

그래서:
- chunk를 더 작게 줄이고
- tokenizer에 넣기 전에 긴 텍스트를 미리 분할하는 안전장치를 넣었다

이건 프롬프트 문제가 아니라 파이프라인 입력 계약 문제로 봤다.

---

## 6. 임베딩 경로와 청킹 모델이 왜 분리되어 있나

이 부분이 오늘 많이 헷갈렸던 포인트다.

### 현재 고정 기준

- 임베딩 생성: 원격 endpoint
- 청킹/token counting: 로컬 `dragonkue/bge-m3-ko`

### 왜 이렇게 나눴나

임베딩은 운영 환경의 서버 정책 문제다.  
하지만 chunk sizing은 전처리 재현성과 inspectability 문제다.

즉:
- 벡터를 어디서 만들지
- 텍스트를 몇 토큰으로 볼지

이 둘은 같은 문제가 아니다.

---

## 7. Part 2에서 제일 중요한 설계 포인트

### 7-1. 왜 BM25 + vector를 같이 쓰는가

BM25는 키워드 일치에 강하다.  
vector는 의미 유사도에 강하다.

OCP 질문은 둘 다 필요하다.

예:
- `etcd 백업은 어떻게 해?`
  -> 용어가 정확한 운영 질문
- `오픈시프트가 뭐야?`
  -> 설명형 broad question

하나만 쓰면 한쪽에서 계속 틀린다.

### 7-2. 왜 query rewrite가 중요하냐

검색은 질문 원문 그대로 던진다고 항상 잘 되지 않는다.

예를 들면:
- `그 복구는 어떻게 해?`
  -> 단독으로는 무슨 복구인지 모름
- `오픈시프트가 뭐야?`
  -> broad intro인지, 특정 기능 질문인지 판단 필요
- `백업 복구 문서는 어디서 봐?`
  -> 절차 질문이 아니라 문서 위치 찾기 질문

즉 query rewrite는 "질문을 멋있게 바꾸는 것"이 아니라  
`검색기에게 맞는 형태로 질문 의도를 정리하는 작업`이다.

### 7-3. 오늘 확인한 교훈

리라이팅을 잘못하면 retrieval 전체를 망친다.

실제 문제:
- `etcd가 왜 중요한지 설명해줘`
  를 예전엔 무조건 `backup restore` 쪽으로 확장해서
  개념 질문이 백업 문서로 쏠렸다.

즉 retrieval이 틀린 게 아니라  
그 전에 rewrite가 질의를 먼저 망치는 경우가 있었다.

### 7-4. 현재 rewrite 철학

질문별 하드코딩이 아니라 의도 기반 분류다.

현재 분류 예시:
- generic intro
- doc-locator
- etcd concept
- etcd ops
- follow-up
- unsupported external product

---

## 8. Hybrid Fusion은 어떻게 작동하나

현재 검색은 대략 이렇게 합친다.

1. BM25 top-k 추출
2. vector top-k 추출
3. RRF로 결합
4. 책 수준 prior 적용

기본 아이디어:

```text
fused_score += weight / (rrf_k + rank)
```

하지만 실제로는 그 위에 추가 보정이 있다.

- BM25와 vector 양쪽에서 같이 지지된 책이면 가산
- 한국어 질의에서 한국어 텍스트면 가산
- broad intro면 `architecture`, `overview` boost
- doc-locator면 `_apis` 계열 감점
- unsupported query는 아예 검색 short-circuit

즉 단순히 BM25 점수와 vector 점수를 섞는 게 아니라  
`랭크 기반 결합 + 도메인 priors` 구조다.

---

## 9. unsupported 질문은 왜 일부러 막나

예:
- `Harbor 설치 방법 알려줘`
- `EKS랑 AKS 요금 비교해줘`

이런 질문은 OCP corpus 밖이다.

여기서 억지로 비슷한 문서를 던져주면 오답이다.  
그래서 지금은 unsupported product를 감지하면 검색을 끊고 warning을 남긴다.

이건 실패가 아니라 정책이다.

즉:
- 못 찾은 게 아니라
- 일부러 안 찾는 것

---

## 10. Part 3에서 제일 중요한 포인트

검색 top-k를 전부 프롬프트에 넣으면 답변이 오염된다.

그래서 context assembly를 따로 둔다.

### 현재 context gate가 하는 일

- top score가 약하면 clarification 유도
- 경쟁 책이 많으면 context를 비움
- 상위 책 위주로만 context 구성
- 책당 너무 많은 청크를 넣지 않음
- 같은 section이 여러 책에 거울처럼 복제된 경우 하나만 남김

즉 retrieval이 "후보 수집"이라면  
context assembly는 "최종 증거 채택" 단계다.

---

## 11. 지금 점수만 믿으면 안 되는 이유

오늘 중요한 교훈 중 하나다.

offline eval 점수가 좋아도,
UI에서 실제 질문을 넣었을 때 결과가 쓰레기면
그 점수는 과적합일 수 있다.

실제 경험:
- retrieval/answer 지표는 좋아 보였는데
- `오픈시프트가 뭐야?`, `etcd 백업은 어떻게 해?` 같은 대표 질문이 망가졌다

그래서 지금 프로젝트는:

- offline eval은 회귀 방지용
- 최종 판단은 실물 QA

라는 원칙으로 간다.

## 13. 지금 남은 진짜 문제

현재 남은 핵심 이슈는 3개다.

### 13-1. 한국어 미지원 페이지 처리 정책

문서는 찾았는데 한국어 본문이 없으면 어떻게 할 것인가?

예상 정책 방향:
- 문서는 있다고 안내
- 한국어 본문이 없다고 명시
- 내부 문서 경로는 제공
- 억지로 한국어 답변을 길게 꾸며내지 않음

### 13-2. citation 내부 링크 일관성

사용자는 citation 클릭 시 내부 `/docs/...` 경로를 기대한다.  
외부 웹 링크처럼 보이거나 404가 나면 폐쇄망 RAG UX가 깨진다.

### 13-3. answer 품질

retrieval은 많이 나아졌지만 answer 정책은 아직 다듬을 여지가 많다.

특히:
- 영어 문서 대응
- unsupported/no-answer 응답
- 애매한 질문 clarification

## 14. 오늘 작업을 통해 배운 점

### 배운 점 1

리라이팅은 retrieval의 보조가 아니라 핵심이다.

### 배운 점 2

점수만 보면 안 된다.  
실사용 질문으로 반드시 깨봐야 한다.

### 배운 점 3

warning과 error는 그냥 넘어가면 안 된다.  
특히 version mismatch, token overflow, input contract violation은 파이프라인 문제로 바로 봐야 한다.

### 배운 점 4

"질문 하나 망함 -> alias 추가" 식으로 고치면 결국 하드코딩 늪에 빠진다.  
질문별 땜질이 아니라 의도 기반 규칙으로 가야 한다.

7. 왜 score가 좋아도 UI 테스트가 더 중요하냐

## 17. 마지막으로 정말 짧게 요약

이 프로젝트는 단순한 챗봇이 아니다.

- Part 1은 문서를 검색 가능하게 만드는 단계
- Part 2는 질문을 문서 검색으로 바꾸는 단계
- Part 3은 찾은 문서만으로 안전하게 답하는 단계
- Part 4는 실사용 QA 단계

그리고 현재 가장 중요한 현실은:

`검색은 많이 안정됐지만, 영어 문서/내부 링크/답변 정책 쪽이 아직 병목이다.`

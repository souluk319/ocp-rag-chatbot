OpenShift ConfigMap 과  Secret
OpenShift 에서  애플리케이션을  운영할  때 , 설정값과 민감 정보를 어떻게 분리해서  관리하
느냐는 매우  중요합니다 . 하드코딩된  환경  변수나  이미지  내부에  포함된  설정  파일은  운영
유연성과 보안성을  모두  떨어뜨립니다 .
이 문서는 OpenShift 에서  사용하는  ConfigMap과 Secret의 개념, 차이점 , 사용  방법 , 예
시 YAML, 운영  시  주의사항까지  한  번에  이해할  수  있도록  정리한  문서입니다 .
1. 왜 ConfigMap 과  Secret 이  필요한가
컨테이너 이미지는  가능하면  불변(immutable) 하게 유지하는  것이  좋습니다 . 즉 , 애플리케
이션 코드는  이미지에  담되 , 실행  환경에  따라  달라지는  값은  외부에서  주입하는  방식이  바
람직합니다 .
예를 들어 다음과  같은  값들은  환경마다  달라질  수  있습니다 .
DB 접속 주소
Redis 호스트명
로그 레벨
외부 API Endpoint
인증서
사용자명/ 비밀번호
토큰, API Key
이런 값을 모두  이미지에  직접  넣어버리면  다음과  같은  문제가  생깁니다 .
개발/검증/ 운영  환경별  분리가  어려움
설정 변경 시  이미지를  다시  빌드해야  함
민감 정보가  이미지나  Git 저장소에  노출될  수  있음
운영 중 설정  추적과  변경  관리가  복잡해짐
OpenShift 에서는  이를  해결하기  위해  다음  두  가지  오브젝트를  사용합니다 .

2. ConfigMap 이란
ConfigMap은 문자열 기반의  일반  설정  데이터를  저장하는  Kubernetes/OpenShift 오브
젝트입니다 .
대표적인 사용  사례는  다음과  같습니다 .
애플리케이션  설정  파일
환경 변수 값
URL, 포트 , 기능  플래그
로그 레벨
비민감성 애플리케이션  옵션
즉, 노출되어도  치명적이지  않은  설정값을 담는 용도라고  이해하시면  됩니다 .
ConfigMap 예시
APP_MODE=production
LOG_LEVEL=info
API_URL=https://api.example.com
application.properties 파일 내용
3. Secret 이란
Secret은 비밀번호 , 토큰 , 인증서 , 키와  같이  민감한 정보를 저장하는  오브젝트입니다 .
대표적인 사용  사례는  다음과  같습니다 .
DB 사용자명 / 비밀번호
TLS 인증서와  개인키
Docker Registry 인증  정보
OAuth 토큰
SSH Key
API Key

Secret은 ConfigMap 과  사용  방식이  유사하지만 , 목적은  분명히  다릅니다 .
중요한 점은  Secret 이  있다고  해서  자동으로  완전한  보안이  보장되는  것은  아니라는  것
입니다. Secret 은  민감  정보용  객체이며 , 접근  권한과  RBAC, audit, etcd 암호화 , 노출
최소화 전략과  함께  운영되어야  합니다 .

5. OpenShift 에서  어떻게  주입하는가
ConfigMap 과  Secret 은  보통  다음  두  가지  방식으로  Pod 에  전달됩니다 .
5.1 환경 변수로  주입
애플리케이션이  환경  변수  기반  설정을  읽는  구조일  때  적합합니다 .
장점:
간단함
대부분의 애플리케이션에서  바로  사용  가능
12-factor app 스타일에  적합
단점:
환경 변수 노출  범위를  잘  관리해야  함

3

대형 설정 파일에는  부적합
5.2 볼륨으로  마운트
설정 파일, 인증서 , 키  파일처럼  파일 형태로 애플리케이션에  전달해야  할  때  적합합니다 .
장점:
설정 파일 구조를  그대로  유지  가능
인증서/키 파일  전달에  적합
특정 경로에  파일  형태로  배치  가능
단점:
애플리케이션이  해당  경로를  읽도록  설계되어야  함
특정값을  환경변수로  주입: 필요한 key 하나씩  골라서  환경변수로  넣음
envFrom으로 주입: ConfigMap/Secret 의  key 들을  한꺼번에  환경변수로  넣음
파일로 마운트 : key-value 를  환경변수가  아니라  파일  형태로  넣음

2. OpenShift 운영  관점에서  중요한  포인트
12.1 값이 바뀌면  Pod 재시작이  필요한  경우가  많다
환경 변수로  주입된  ConfigMap/Secret 값은  일반적으로  실행 중인  컨테이너  내부에서  자
동 반영되지  않습니다 .
따라서 값 변경  후에는  보통  다음  중  하나가  필요합니다 .
Deployment rollout 재시작
새로운 Pod 재생성
애플리케이션  자체의  설정  리로드  로직
예:
oc rollout restart deployment/myapp -n my-project
12.2 볼륨  마운트  방식은  애플리케이션  구조를  봐야  한다
파일이 갱신되어도  애플리케이션이  해당  파일을  다시  읽지  않으면  의미가  없습니다 .
즉, Kubernetes/OpenShift 기능만  보는  것이  아니라  애플리케이션이  설정  리로드를  지원
하는지도 함께 확인해야  합니다 .

# OCP RAG v2 실행 기준 요약

이 문서는 매번 작업 시작 전에 먼저 보는 **실행용 고정 컨텍스트 요약본**이다.  
상세 기준은 [fixed-context.md](/C:/Users/soulu/cywell/ocp-rag-chatbot/docs/v2/fixed-context.md)에 있다.

## 1. 과제의 본질

이 과제는 보여주기식 MVP를 만드는 과제가 아니다.  
이 과제는 **제대로 작동하는 OCP RAG 파이프라인을 설계하고, 그 구조와 동작을 설명할 수 있는지**를 평가한다.

즉, 다음을 증명해야 한다.

1. 공식 `openshift-docs`를 어떻게 수집하는지
2. `.adoc`를 어떻게 정규화하고 메타데이터를 붙이는지
3. OpenDocuments 기준 RAG 구조 위에서 검색과 응답이 어떻게 이어지는지
4. citation이 어떻게 붙고, 클릭 시 HTML 원문이 어떻게 열리는지
5. 멀티턴, 세션, 폐쇄망 반입 구조까지 설명 가능한지

## 2. 절대 규칙

1. **알아서 판단하지 않는다.**
2. 해야 할 일이 명확하지 않으면 사용자에게 되묻는다.
3. 확신이 서기 전까지 추가 질문을 이어가서 할 일을 명확하게 만든다.
4. 사용자가 방향을 못 잡으면 옵션을 줘서 선택하게 한다.
5. 절대로 멋대로 해석한 결과물을 만들지 않는다.
6. 직접 재현해서 증명한 것만 “된다”고 말한다.
7. `localhost:8000`에서 직접 확인 가능한 결과를 우선한다.
8. README와 보고 문서는 실제 상태를 과장하지 않는다.

## 3. 허용 / 비허용

### 허용

1. guardrail, rescue, fallback 같은 운영 안전장치
2. 단, 어디까지가 raw retrieval이고 어디부터 안전장치인지 구분해서 설명할 것

### 비허용

1. 특정 질문만 빠르게 통과시키는 로직을 전체 RAG 성능처럼 포장하기
2. 내부 지표만 맞고 실제 UI가 깨진 상태를 “완료”라고 부르기
3. 문서와 체크리스트만 예쁘게 만들고 실제 런타임 검증을 빼먹기

## 4. 성공 기준

다음이 모두 맞아야 “성공”이라고 말할 수 있다.

1. `localhost:8000`에서 직접 동작한다.
2. 한국어 질문에 한국어 답변이 나온다.
3. citation이 붙는다.
4. citation 클릭 시 HTML 문서가 열린다.
5. 답변 경로를 설명할 수 있다.
6. active index / active manifest / active profile이 실제 런타임과 일치한다.

## 5. 실제 6명 서브에이전트

1. `Harvey`: 사용자 질문/시나리오 검증
2. `Turing`: 한국어 답변 품질 검증
3. `Peirce`: 문서/메타데이터/출처 정합성 검증
4. `Carver`: 테스트/평가/pass-fail 기준 검증
5. `Banach`: `localhost:8000` 직접 확인 체크포인트 설계
6. `Bohr`: 과장 방지, README/보고 문구 감수

## 6. 작업 전 체크 질문

1. 지금 하려는 일이 파이프라인 설계 능력 증명과 직접 연결되는가?
2. 무엇을 만들지 아직 애매한가?
3. 애매하다면 사용자에게 다시 물어야 하는가?
4. 지금 결과를 사용자가 `localhost:8000`에서 직접 확인할 수 있는가?

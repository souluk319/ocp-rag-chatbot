# OCP RAG v2 실행용 고정 컨텍스트

이 문서는 **매번 작업 시작 전에 먼저 보는 실행 기준서**다.

- 길게 고민하기 전에 이 문서를 먼저 본다.
- 이 문서와 실제 작업이 어긋나면 작업을 멈추고 방향부터 바로잡는다.
- 더 자세한 설명이 필요할 때만 [fixed-context.md](/C:/Users/soulu/cywell/ocp-rag-chatbot/docs/v2/fixed-context.md)를 본다.

## 1. 절대 목표

우리는 문서 정리 프로젝트를 하는 게 아니다.  
우리는 **폐쇄망에서 실제로 쓸 수 있는 OCP 운영 도우미 챗봇**을 만든다.

반드시 만족해야 하는 핵심:

1. 사용자는 한국어로 질문한다.
2. 챗봇은 공식 `openshift-docs` 근거로 답한다.
3. 답변에는 출처가 붙는다.
4. 출처를 누르면 HTML 원문이 열린다.
5. 멀티턴 문맥이 유지된다.
6. `localhost:8000` 에서 실제로 테스트 가능해야 한다.

## 2. 오늘 우선순위

방대한 데이터보다 **제대로 도는 최소 파이프라인**이 먼저다.

우선순위는 항상 이 순서다.

1. 기본 질문이 한국어로 답되는가
2. UI와 스트리밍 한글이 안 깨지는가
3. citation 이 보이고 클릭하면 문서가 열리는가
4. 실제 런타임이 실제 코퍼스를 보고 있는가
5. 그 다음에 retrieval 품질과 코퍼스 확장

## 3. 절대 잊지 말 것

1. 임베더 `BAAI/bge-m3` 는 검색 품질용이다. 생성 한글 품질을 자동으로 해결하지 않는다.
2. `Qwen/Qwen3.5-9B` 는 한국어 출력이 흔들릴 수 있으니 답변 shaping 이 필요하다.
3. citation 이 맞지 않으면 챗봇은 실패다.
4. 검증 도구가 아니라 **사용자 눈에 보이는 결과**가 진짜 완료 기준이다.
5. 오래 걸리는 명령은 실행 전에 목적과 이유를 먼저 설명해야 한다.

## 4. 하면 안 되는 일

1. 문서와 검증 도구만 쌓고 실제 사용자 문제를 미루지 않는다.
2. minor 버전이 안 정해졌는데 파이프라인을 특정 버전에 하드고정하지 않는다.
3. `.env` 비밀값을 커밋하지 않는다.
4. “대충 된다”를 완료로 말하지 않는다.
5. 검은 콘솔창을 여러 개 띄워서 사용자를 헷갈리게 하지 않는다.

## 5. 실제 완료 기준

아래 다섯 개가 맞아야 “됐다”고 말한다.

1. `오픈시프트가 뭐야`, `OCP가 뭐야` 에 한국어 답변이 나온다.
2. 답변에 citation 이 1개 이상 붙는다.
3. citation 클릭 시 HTML 문서가 열린다.
4. `localhost:8000` 에서 직접 재현 가능하다.
5. 멀티턴 follow-up 이 최소한 기본 시나리오에서 유지된다.

## 6. 6명 서브작업자 요약

실제로 병렬 작업을 쓸 때는 아래 역할로 나눈다.

1. `Ramanujan`
   역할: 질문 해석, query classification, basic question rescue
2. `Arendt`
   역할: 한국어 답변 표현, UI 문구, 사용자 경험
3. `Peirce`
   역할: 공식 문서 범위, source profile, corpus contamination 방지
4. `Gibbs`
   역할: gateway, bridge, OpenDocuments 연동, 런타임 고정
5. `Feynman`
   역할: 재현 테스트, smoke, regression, 실패 확인
6. `Russell`
   역할: 구조 정리, README, 저장소 맵, 문서-코드 정합성

## 7. 작업 전 체크 질문

작업 시작 전에 이 네 가지를 본다.

1. 지금 고치려는 게 사용자에게 바로 보이는 문제인가?
2. 지금 고치려는 게 citation / 한국어 / 런타임 중 하나와 직접 연결되는가?
3. 긴 명령이라면 왜 오래 걸리는지 설명할 수 있는가?
4. 끝나고 나면 내가 `localhost:8000` 에서 직접 확인할 수 있는가?

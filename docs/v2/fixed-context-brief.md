# OCP RAG v2 실행 기준서 요약

이 문서는 매번 작업 시작 전에 먼저 보는 **실행용 기준 문서**다.
상세 설명이 필요할 때만 [fixed-context.md](/C:/Users/soulu/cywell/ocp-rag-chatbot/docs/v2/fixed-context.md)를 본다.

## 1. 절대 목표

이 과제의 목표는 **보여주기용 MVP나 빠른 데모를 만드는 것**이 아니다.
이 과제의 핵심은 **제대로 작동하는 OCP RAG 파이프라인을 설계하고, 그 구조를 설명할 수 있는 능력**을 증명하는 것이다.

즉 우리가 보여줘야 하는 것은:

1. 공식 `openshift-docs` 원문을 어떻게 수집하는가
2. `.adoc`를 어떻게 정규화하고 메타데이터를 붙이는가
3. OpenDocuments 기준 RAG 구조 위에서 어떻게 검색/응답을 구성하는가
4. 왜 그 답변이 나왔는지 citation으로 설명할 수 있는가
5. citation을 누르면 실제 HTML 원문이 열리는가
6. 한국어 질문, 멀티턴, 폐쇄망 운영 구조까지 일관되게 설명 가능한가

## 2. 지금의 성공 기준

아래가 모두 맞아야 “됐다”고 말할 수 있다.

1. `localhost:8000`에서 직접 동작한다.
2. 한국어 질문에 한국어 답이 나온다.
3. 답변에 citation이 붙는다.
4. citation 클릭 시 HTML 문서가 열린다.
5. 답변 경로가 무엇인지 설명 가능하다.
6. raw retrieval / policy-prepared / rescue 경로를 구분해서 설명할 수 있다.
7. README와 보고서가 실제 상태를 과장하지 않는다.

## 3. 절대 금지

1. 특정 심사 질문만 맞추는 하드코딩을 핵심 성과처럼 포장하지 않는다.
2. rule-based fast path를 “순수 RAG 성능”처럼 말하지 않는다.
3. 내부 검증 통과만으로 완료라고 말하지 않는다.
4. `localhost:8000`에서 직접 확인 못 한 것은 “된다”고 쓰지 않는다.
5. README를 내부 작업일지처럼 쓰지 않는다.

## 4. 허용되는 안전장치

운영형 guardrail이나 rescue는 넣을 수 있다.
단, 아래 조건을 만족해야 한다.

1. 어디까지가 RAG 경로이고 어디부터가 안전장치인지 구분 가능해야 한다.
2. 안전장치가 raw retrieval 문제를 가리는 용도로 쓰이면 안 된다.
3. 보고서와 README에서 과장 없이 설명 가능해야 한다.

## 5. 지금의 우선순위

1. 파이프라인이 실제로 도는지 확인
2. 기본 질문과 핵심 운영 질문이 실제로 통과하는지 확인
3. retrieval 품질과 citation 정합성 확인
4. 그 다음에만 데이터 확장과 추가 최적화

## 6. 실제 6명 서브에이전트

1. `Harvey`: 질문 시나리오와 사용자 관점 검토
2. `Turing`: 한국어 응답 품질과 표현 검토
3. `Peirce`: 공식 문서/메타데이터/출처 정합성 검토
4. `Carver`: 평가 스크립트와 pass/fail 기준 검토
5. `Banach`: localhost 기준 직접 확인 포인트 설계
6. `Bohr`: 과장 금지, 문서/보고 문구 감수

## 7. 매 작업 전 확인 질문

1. 이 작업이 파이프라인 설계 능력 증명에 직접 연결되는가?
2. 이 작업이 단순 데모 연출이 아니라 실제 동작 개선인가?
3. 끝나면 `localhost:8000`에서 직접 확인 가능한가?
4. 결과를 상사/심사자에게 정직하게 설명할 수 있는가?

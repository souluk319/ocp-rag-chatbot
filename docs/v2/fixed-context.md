# OCP RAG v2 고정 컨텍스트

이 문서는 이 프로젝트의 **최상위 기준 문서**입니다.  
우선순위가 흔들리거나, 다음 작업이 무엇인지 불명확해지거나, 결과물이 실제 목표에서 벗어날 때 이 문서로 돌아옵니다.

## 1. 과제의 본질

이 과제는 보여주기용 MVP를 만드는 과제가 아닙니다.  
이 과제는 **실제로 작동하는 OCP RAG 파이프라인을 설계하고, 그 구조와 동작을 설명할 수 있는지**를 평가합니다.

즉, 다음을 증명해야 합니다.

1. 공식 문서를 어떻게 수집하는가
2. `.adoc`를 어떻게 정규화하는가
3. 메타데이터와 문서 연결 관계를 어떻게 보존하는가
4. `OpenDocuments` 기준 RAG 구조에 어떻게 연결하는가
5. retrieval, 응답 생성, citation, click-through가 어떻게 이어지는가
6. 멀티턴과 폐쇄망 반입/갱신 구조를 어떻게 설명하는가

## 2. 우리의 최종 목표

우리가 만들려는 것은 다음 조건을 만족하는 **OCP 운영 가이드 챗봇**입니다.

1. 공식 Red Hat/OpenShift 문서를 근거로 답한다.
2. OCP 운영 질문에 한국어로 답한다.
3. 답변마다 citation이 붙는다.
4. citation을 클릭하면 HTML 원문 문서가 열린다.
5. 실제로 어떤 파이프라인 단계를 거쳤는지 UI에서 확인할 수 있다.
6. 결과가 정확도 기준을 만족한다.

## 3. 절대 행동 규칙

아래 규칙은 예외 없이 지킵니다.

1. 알아서 판단하지 않는다.
2. 목표, 범위, 완료 기준이 모호하면 사용자에게 먼저 묻는다.
3. 추가 질문으로 의도를 명확히 한 뒤에만 작업한다.
4. 내가 이해한 목표/범위/완료 기준을 다시 보고하고, 사용자 확인을 받은 뒤에만 작업을 시작한다.
5. 사용자가 방향을 잡기 어렵다면 옵션을 제시하고 선택하게 한다.
6. 절대로 멋대로 해석한 결과물을 만들지 않는다.
7. 재현해서 증명하지 못한 기능은 완료라고 말하지 않는다.
8. README나 보고 문서에 실제 상태를 과장하지 않는다.

## 5. 데이터 순도와 코퍼스 설계 원칙

### 5.1 데이터 순도

1. 목표는 **공식 Red Hat/OpenShift 문서로 OCP 가이드를 안내하는 챗봇**이다.
2. 따라서 `ROSA`, `OSD`, `OCM`, `MicroShift`, `managed service` 계열 문서는 기본적으로 경계 대상이다.
3. 하지만 무조건 삭제가 아니라, **정확도 향상에 실제로 도움이 되는지**를 평가한 뒤에 포함/제외를 결정한다.

### 5.2 modules 관계

`modules/`는 단순 제외 대상이 아니다.

우리가 반드시 지켜야 할 설계 원칙:

1. 상위 가이드 `.adoc`의 `include::modules/...` 관계를 읽는다.
2. `xref:` 로 연결되는 인접 가이드 관계를 읽는다.
3. canonical document model은 **상위 가이드 + 포함된 module + 참조 관계**를 함께 표현해야 한다.
4. 검색과 citation은 상위 가이드와 실제 포함된 절차 조각을 둘 다 추적할 수 있어야 한다.
5. 절차형 답변이 끊기지 않게, chunk는 이 canonical 관계를 보존해야 한다.

## 6. Golden Set 100

Golden Set은 코퍼스와 파이프라인을 평가하는 절대 기준이다.

### 6.1 위치

1. 질문 세트 파일: [C:\Users\soulu\cywell\ocp-rag-chatbot\eval\benchmarks\golden_set_100.csv](/C:/Users/soulu/cywell/ocp-rag-chatbot/eval/benchmarks/golden_set_100.csv)
2. 설명 문서: [C:\Users\soulu\cywell\ocp-rag-chatbot\docs\v2\golden-set-4.20.md](/C:/Users/soulu/cywell/ocp-rag-chatbot/docs/v2/golden-set-4.20.md)

### 6.2 구성 원칙

1. 한국어 질문 기준
2. 설치/업데이트/운영 질문 비중을 높임
3. Disconnected, 복합 설정, 장애 상황 등 까다로운 질문 포함
4. 범위 밖 질문 포함
5. 각 문항은 다음을 가진다.
   - 질문
   - 의도/카테고리
   - 기대 출처 범위
   - 기대 정답 형태

### 6.3 Success Criteria

1. 검색 재현율(Recall): 90% 이상
2. 답변 정확도(Accuracy): 85% 이상
3. 환각율(Hallucination): 0%
4. 출처 연결 정확도: 100%

## 7. 허용되는 안전장치와 금지 사항

### 허용

guardrail, rescue, fallback, policy-prepared rerank는 **안전장치**로 사용할 수 있다.

단, 아래 조건이 붙는다.

1. 어디까지가 raw retrieval인지 설명 가능해야 한다.
2. 어디부터가 안전장치인지 설명 가능해야 한다.
3. UI와 trace에서 실제 경로를 숨기지 않아야 한다.

### 금지

1. 특정 질문만 빠르게 통과시키는 경로를 전체 파이프라인 성능처럼 포장하기
2. trace나 UI만 그럴듯하게 만들고 실제 파이프라인 품질을 방치하기
3. 내부 지표만 좋고 `localhost:8000`에서 직접 보면 망가지는 상태를 통과로 간주하기
4. README에 내부 작업 기록이나 과장된 상태를 적기

## 9. 작업 시작 순서

모든 작업은 아래 순서를 따른다.

1. `fixed-context-brief.md` 확인
2. 이번 작업이 파이프라인 설계 능력 증명과 직접 연결되는지 확인
3. 모호하면 사용자에게 질문
4. 내가 이해한 목표/범위/완료 기준을 다시 보고
5. 사용자 확인을 받은 뒤 실제 작업 시작
6. 구현
7. 테스트
8. 검증
9. 정리
10. `localhost:8000` 또는 평가 자산 기준으로 재현 가능하게 보고

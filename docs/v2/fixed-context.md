# OCP RAG v2 고정 컨텍스트

이 문서는 이 프로젝트의 **절대 기준 문서**다.
우선순위가 흔들리거나, 작업이 데모 연출 쪽으로 새거나, “지금 우리가 뭘 증명해야 하지?”가 헷갈릴 때 이 문서로 돌아온다.

짧은 실행 기준은 [fixed-context-brief.md](/C:/Users/soulu/cywell/ocp-rag-chatbot/docs/v2/fixed-context-brief.md)에 있다.
이 문서는 그 기준의 상세판이다.

## 1. 과제의 본질

이 과제는 예쁜 데모나 보여주기용 MVP를 만드는 과제가 아니다.
이 과제는 **RAG에서 LLM으로 이어지는 파이프라인을 어떻게 설계하고, 공식 문서를 어떻게 다루고, 왜 이 구조가 맞는지 설명할 수 있는가**를 평가하는 과제다.

즉 심사자가 보고 싶은 것은 아래다.

1. 공식 문서 원천을 어떻게 수집하는가
2. 문서를 어떤 중간 구조로 정규화하는가
3. 어떤 메타데이터를 붙이고 왜 필요한가
4. retrieval이 어떤 기준으로 동작하는가
5. 답변 생성 전에 어떤 context가 구성되는가
6. citation이 어떻게 만들어지고 왜 신뢰할 수 있는가
7. 폐쇄망 반입/갱신 구조까지 설명 가능한가

핵심 문장:

**우리는 “답이 빨리 나오는 챗봇”이 아니라, “제대로 작동하는 파이프라인을 설계하고 설명 가능한 챗봇”을 만들어야 한다.**

## 2. 프로젝트의 최종 목표

우리가 만드는 것은 다음 조건을 만족하는 OCP 운영 도우미 챗봇이다.

1. `openshift-docs`를 공식 원천 데이터로 사용한다.
2. RAG 구조는 OpenDocuments 기준으로 통일한다.
3. 사용자는 한국어로 질문한다.
4. 답변은 한국어 중심으로 제공한다.
5. 답변에는 출처가 반드시 포함된다.
6. 출처를 클릭하면 HTML 원문 문서가 열린다.
7. 멀티턴 문맥을 유지한다.
8. 폐쇄망 반입/재인덱싱/활성화/롤백까지 설명 가능한 구조를 가진다.

## 3. 설계상 반드시 지켜야 할 것

### 3.1 데이터 원천

1. 공식 원천은 `openshift-docs`다.
2. 원문 포맷은 `.adoc`이다.
3. 웹/PDF는 배포본이고, RAG용 원천은 `.adoc` 기반 정규화 결과다.
4. 특정 minor 버전이 정해지지 않은 현재는 validation-mode로 가되, 나중에 source profile로 전환 가능해야 한다.

### 3.2 파이프라인

우리가 설명할 수 있어야 하는 핵심 파이프라인은 아래다.

1. source mirror / source profile 선택
2. `.adoc` 수집
3. 정규화
4. canonical document model 생성
5. 검색용 텍스트 / chunk 생성
6. HTML citation view 생성
7. metadata manifest 생성
8. OpenDocuments 기준 retrieval / answer path 연결
9. citation click-through

### 3.3 답변 경로 투명성

답변은 하나의 길로만 간다고 가정하면 안 된다.
현재 구조상 아래 경로를 구분해서 설명할 수 있어야 한다.

1. raw retrieval path
2. policy-prepared path
3. rescue / guardrail path

이 세 경로는 섞어 말하면 안 된다.
특히 rescue path를 raw RAG 성능처럼 포장하면 안 된다.

### 3.4 현재 모델 기준

1. 생성 모델: `Qwen/Qwen3.5-9B`
2. 임베딩 baseline: `BAAI/bge-m3`
3. 임베딩 차원: `1024`

중요한 전제:

- `BAAI/bge-m3`는 retrieval 개선을 위한 선택이지, 한국어 생성 품질을 자동으로 해결해 주는 해결책이 아니다.
- 생성 품질 문제와 retrieval 품질 문제는 따로 다뤄야 한다.

## 4. 절대 금지 사항

### 4.1 데모 포장 금지

아래는 절대 하면 안 된다.

1. 특정 질문 몇 개만 빠르게 통과시키는 로직을 전체 파이프라인 성과처럼 말하기
2. rule-based fast path를 “RAG가 좋아서 빠르다”고 설명하기
3. citation이 열린다는 이유로 retrieval 오류를 덮기
4. 문서/체크리스트/평가 스크립트만 쌓고 실제 `localhost:8000` 상태를 확인하지 않기
5. 심사자가 조금만 파보면 무너질 설명을 README나 보고서에 쓰기

### 4.2 상태 과장 금지

1. 직접 재현하지 못한 것은 “동작한다”고 쓰지 않는다.
2. `localhost:8000`에서 직접 안 되는 것은 완료라고 부르지 않는다.
3. production-ready, fully ready 같은 표현은 실제로 증명했을 때만 쓴다.

## 5. 허용되는 운영형 안전장치

운영형 guardrail, rescue, fallback은 사용할 수 있다.
다만 아래 조건을 모두 만족해야 한다.

1. 목적이 명확해야 한다.
2. 어떤 질문군에 왜 넣었는지 설명 가능해야 한다.
3. raw retrieval 성능과 분리해서 보고해야 한다.
4. README/보고서에서 과장 없이 적어야 한다.
5. 심사자가 물었을 때 “이건 운영형 안전장치이고, 순수 retrieval 성능과는 다르다”고 바로 설명할 수 있어야 한다.

즉 안전장치는 가능하지만, **핵심 파이프라인 능력의 증거를 가리는 용도**로 쓰면 안 된다.

## 6. 지금의 성공 기준

이 프로젝트에서 “성공”이라고 말하려면 아래를 만족해야 한다.

1. `localhost:8000`에서 직접 열 수 있다.
2. 기본 질문이 한국어로 답된다.
3. 핵심 운영 질문이 한국어로 답된다.
4. citation이 붙는다.
5. citation 클릭 시 HTML 원문이 열린다.
6. 답변 경로를 설명할 수 있다.
7. active index / active manifest / active source profile이 실제 런타임과 일치한다.
8. README와 내부 문서가 실제 상태를 과장하지 않는다.

## 7. 가장 중요한 현재 리스크

현재 가장 위험한 것은 아래다.

1. widened corpus에서 raw retrieval baseline이 약하다.
2. 일부 질문이 rescue path에 의존한다.
3. 사용자가 보기에 빠르게 답하는 것과 실제 파이프라인 품질이 다를 수 있다.
4. README나 시연 설명이 실제 상태보다 앞서가면 심사에서 바로 걸린다.

따라서 현재 우선순위는:

1. 실제 파이프라인이 도는지
2. 실제 사용자 질문이 통과하는지
3. 그 답변 경로를 정직하게 설명할 수 있는지

순서로 잡아야 한다.

## 8. 실제 6명 서브에이전트

이 프로젝트는 아래 6명을 고정 역할로 본다.

### 8.1 Harvey

- 역할: 데모 질문/사용자 시나리오 설계
- 책임:
  - 어떤 질문을 실제 확인 질문으로 쓸지 정리
  - 사용자가 어디서 불안해할지 먼저 찾기
  - `localhost:8000` 기준 체크포인트 정리

### 8.2 Turing

- 역할: 한국어 답변 품질 감수
- 책임:
  - 한국어 문장 품질
  - 검색기 말투 제거
  - 과한 영어 인용/경고 문구 제어

### 8.3 Peirce

- 역할: 공식 문서/메타데이터/출처 정합성 검증
- 책임:
  - 어떤 citation family가 맞는지 확인
  - source profile / manifest / lineage 점검
  - contamination 리스크 점검

### 8.4 Carver

- 역할: 평가기/게이트 기준 검토
- 책임:
  - 단계별 pass/fail 기준 검토
  - 과장된 판정 방지
  - smoke / regression / readiness gate 정리

### 8.5 Banach

- 역할: 실제 사용자 확인 경로 검증
- 책임:
  - `localhost:8000` 기준 최소 수동 확인 포인트 설계
  - “직접 확인 가능한가?” 기준 유지

### 8.6 Bohr

- 역할: 보고 문구/과장 방지/정리 감수
- 책임:
  - README
  - 보고서 문구
  - “demo-ready”와 “production-ready” 경계 유지

## 9. 작업 방식 고정 규칙

모든 단계는 아래 순서를 따른다.

1. fixed-context-brief 확인
2. 실제 6명 roster 보고
3. 구현
4. 테스트
5. 검증
6. 정리
7. `localhost:8000` 체크포인트 보고

또한:

1. 10분 이상 걸리는 명령은 이유를 먼저 설명한다.
2. 검은 콘솔창을 여러 개 띄워 사용자를 헷갈리게 하지 않는다.
3. 내부 지표보다 사용자가 직접 볼 수 있는 결과를 우선한다.
4. README는 상사/심사자용 문서로 유지한다.

## 10. 매 작업 전 자기 점검 질문

작업 전에 반드시 아래를 스스로 묻는다.

1. 이 작업이 파이프라인 설계 능력 증명에 직접 연결되는가?
2. 이 작업이 단순 데모 연출이 아니라 실제 동작 개선인가?
3. 끝나면 `localhost:8000`에서 사용자가 직접 확인 가능한가?
4. 이 결과를 심사자에게 정직하게 설명할 수 있는가?

# OCP RAG v2 고정 컨텍스트

이 문서는 이 프로젝트의 **최상위 행동 기준 문서**다.  
우선순위가 흔들리거나, 작업이 데모 위주로 흐르거나, 무엇을 증명해야 하는지 헷갈릴 때 이 문서로 돌아온다.

## 1. 과제 목적

이 과제는 보여주기용 MVP를 만드는 과제가 아니다.  
이 과제는 **실제로 작동하는 OCP RAG 파이프라인을 설계하고, 그 구조와 동작을 설명할 수 있는지**를 평가한다.

따라서 다음이 핵심 평가 대상이다.

1. 공식 문서 수집 구조
2. `.adoc` 정규화 구조
3. 메타데이터 설계
4. OpenDocuments 기준 RAG 구조 연결
5. retrieval, 정책 정렬, 응답 조합
6. citation과 HTML click-through
7. 멀티턴과 세션 메모리
8. 폐쇄망 반입 / 활성화 / 롤백 설명 가능성

## 2. 절대 행동 규칙

다음 규칙은 예외 없이 지킨다.

1. **알아서 판단하지 않는다.**
2. 해야 할 일이 명확하게 플랜이 서지 않을 때에는 사용자에게 되묻는다.
3. 확신이 서기 전까지는 추가 질문을 이어나가서 할 일을 명확하게 만들고 나서 일을 한다.
4. 사용자가 방향을 못 잡으면 옵션을 줘서 선택하게 한다.
5. 절대로 멋대로 해석한 결과물을 만들지 않는다.
6. 내가 이해한 작업 목표, 범위, 완료 기준을 사용자에게 다시 짧게 보고하고, **이해가 맞는지 확인된 뒤에만** 작업을 시작한다.
7. 지시가 모호할 때는 작업보다 질문이 먼저다.
8. 직접 재현해서 증명하지 못한 것은 “된다”고 말하지 않는다.
9. 내부 지표만 맞는 상태를 완료라고 부르지 않는다.
10. `localhost:8000`에서 사용자가 직접 확인 가능한 결과를 우선한다.
11. README와 보고 문서는 실제 상태를 과장하지 않는다.

## 3. 프로젝트의 최종 목표

우리가 만들려는 것은 다음 조건을 만족하는 OCP 운영 도우미 챗봇이다.

1. 공식 원천은 `openshift-docs`이다.
2. RAG 구조는 `OpenDocuments` 기준으로 통일한다.
3. 사용자는 한국어로 질문한다.
4. 답변은 한국어 중심으로 제공한다.
5. 답변에는 citation이 반드시 붙는다.
6. citation 클릭 시 HTML 원문 문서가 열린다.
7. 멀티턴 문맥이 유지된다.
8. 폐쇄망 반입/갱신 구조를 설명할 수 있다.

## 4. 기술 기준

### 데이터 원천

1. 공식 원천: `openshift-docs`
2. 원문 형식: `.adoc`
3. 배포본 PDF가 아니라, 원문 `.adoc` 기준으로 수집/정규화한다.

### RAG 기준

1. RAG 구조 기준: `OpenDocuments`
2. 생성 모델: `Qwen/Qwen3.5-9B`
3. 임베딩 baseline: `BAAI/bge-m3`
4. 임베딩 차원: `1024`

### 파이프라인 기본 흐름

1. source profile 선택
2. `.adoc` 수집
3. 정규화
4. canonical document model 생성
5. 검색용 텍스트 / chunk 생성
6. HTML citation view 생성
7. metadata manifest 생성
8. OpenDocuments 기준 retrieval / answer path 연결
9. citation click-through

## 5. 허용되는 안전장치와 금지사항

### 허용

운영 안전장치 자체는 허용된다.

예:

1. guardrail
2. rescue
3. fallback
4. policy-prepared rerank

단, 아래 조건을 지켜야 한다.

1. 어디까지가 raw retrieval인지 구분해서 설명할 것
2. 어디부터 안전장치인지 구분해서 설명할 것
3. 안전장치를 전체 RAG 성능처럼 포장하지 말 것

### 금지

1. 특정 질문만 빠르게 통과시키는 로직을 전체 파이프라인 성능처럼 포장하기
2. 내부 평가 문서만 좋아 보이게 만들고 실제 런타임을 방치하기
3. UI에서 깨지거나 막히는데도 “단계 통과”라고 보고하기
4. README에 내부 진행 문서를 섞어 넣기

## 6. 현재 성공 기준

다음 조건이 맞아야 “작동한다”고 말할 수 있다.

1. `localhost:8000`에서 직접 실행된다.
2. 한국어 질문에 한국어 답변이 나온다.
3. citation이 붙는다.
4. citation 클릭 시 HTML 문서가 열린다.
5. 현재 답변 경로를 설명할 수 있다.
6. active index / active manifest / active source profile이 실제 런타임과 일치한다.

## 7. 현재 핵심 리스크

현재 특히 경계해야 하는 것은 다음이다.

1. widened corpus에서 raw retrieval baseline이 약한 점
2. rescue/guardrail이 raw RAG 성능처럼 보일 위험
3. README나 보고서가 실제 상태보다 앞서 나갈 위험
4. 사용자가 직접 보는 UI 기준과 내부 문서 기준이 어긋나는 위험

## 8. 실제 6명 서브에이전트

### Harvey
- 역할: 사용자 질문/데모 시나리오 검증
- 책임:
  - 사용자가 실제로 물을 질문 정리
  - `localhost:8000` 기준 수동 체크 항목 검토

### Turing
- 역할: 한국어 답변 품질 검증
- 책임:
  - 어색한 한국어, 검색기 말투, 과한 영어 직인용 점검

### Peirce
- 역할: 문서/메타데이터/출처 정합성 검증
- 책임:
  - source profile, manifest, lineage, citation family 검토

### Carver
- 역할: 테스트와 pass/fail 기준 검증
- 책임:
  - 단계별 평가 기준과 테스트 스크립트 검토

### Banach
- 역할: 사용자 눈 기준 검증 설계
- 책임:
  - `localhost:8000`에서 직접 확인 가능한 최소 체크포인트 설계

### Bohr
- 역할: 과장 방지와 문서 감수
- 책임:
  - README, 보고서, 단계 완료 문구가 실제 상태를 과장하지 않게 검토

## 9. 작업 시작 순서

모든 작업은 아래 순서를 따른다.

1. `fixed-context-brief.md` 읽기
2. 이번 작업이 정말 파이프라인 설계 능력 평가와 연결되는지 확인
3. 모호하거나 확신이 없으면 사용자에게 추가 질문
4. 내가 이해한 목표, 범위, 완료 기준을 짧게 다시 보고
5. 사용자 확인 후 실제 6명 서브에이전트 roster와 역할 보고
6. 구현
7. 테스트
8. 검증
9. 정리
10. `localhost:8000` 직접 확인 포인트 보고

## 10. 작업 전 자가 질문

작업 전에 반드시 스스로 묻는다.

1. 지금 하려는 일이 파이프라인 설계 능력 증명과 직접 연결되는가?
2. 해야 할 일이 아직 모호한가?
3. 모호하다면 사용자에게 다시 물었는가?
4. 사용자에게 옵션을 줘야 하는 상황인가?
5. 내가 이해한 내용을 먼저 다시 보고했는가?
6. 지금 결과를 사용자가 `localhost:8000`에서 직접 확인할 수 있는가?

## 11. 최종 한 줄 기준

**이 프로젝트는 “그럴듯한 데모”가 아니라, 실제로 작동하는 OCP RAG 파이프라인을 설계하고 설명 가능한 상태로 만드는 것이 목표다.**

# OCP RAG v2 고정 컨텍스트

이 문서는 **이 프로젝트의 절대 기준 문서**다.  
작업 중 맥락이 흔들리거나, 우선순위가 흐려지거나, 문서가 너무 많아 무엇을 먼저 봐야 할지 혼란스러울 때는 **반드시 이 문서부터 다시 읽는다.**

이 문서의 목적은 세 가지다.

1. 프로젝트의 목표를 잊지 않게 한다.
2. 6명의 서브에이전트가 무엇을 맡아야 하는지 고정한다.
3. 심사까지 남은 시간 동안 무엇을 먼저 고쳐야 하는지 흔들리지 않게 만든다.

이 문서는 제품 방향, 역할 분담, 작업 우선순위, 완료 기준을 한 번에 고정한다.

## 1. 프로젝트의 절대 목표

우리가 만드는 것은 단순한 데모가 아니다.

**우리는 폐쇄망에서 실제로 사용할 수 있는 OCP 운영 도우미 챗봇을 만든다.**

이 목표를 더 정확하게 풀면 아래와 같다.

1. 사용자는 한국어로 질문한다.
2. 챗봇은 공식 OCP 문서를 근거로 답한다.
3. 답변에는 출처가 붙는다.
4. 출처를 누르면 HTML 원문이 열린다.
5. 멀티턴 문맥이 유지된다.
6. 폐쇄망 반입, 재인덱싱, 활성화, 롤백 흐름을 설명할 수 있어야 한다.
7. 특정 minor 버전이 아직 정해지지 않았더라도, 나중에 버전을 갈아끼울 수 있는 파이프라인 구조를 유지해야 한다.

## 2. 이번 심사에서 반드시 통과해야 하는 핵심 요구

이번 심사에서 가장 먼저 걸리는 항목은 아래다.

1. `오픈시프트가 뭐야`, `OCP가 뭐야` 같은 기본 질문에도 최소한의 공식 근거 답변이 나와야 한다.
2. 한글이 깨지면 안 된다.
3. 출처 링크가 실제로 열려야 한다.
4. 답변이 문서와 동떨어진 일반 상식 답변으로 흐르면 안 된다.
5. 챗봇이 실제로 켜져 있어야 한다.
6. 심사자가 바로 눌러볼 수 있는 `localhost:8000` 경로가 안정적으로 살아 있어야 한다.

## 3. 절대 잊지 말아야 할 기술적 사실

### 3.1 모델과 임베딩

1. 생성 모델은 회사 서버의 `Qwen/Qwen3.5-9B` 이다.
2. 임베딩 baseline 은 `BAAI/bge-m3`, `1024` 차원이다.
3. 임베딩을 BGE-M3로 맞췄다고 해서 한국어 출력 품질 문제가 자동으로 해결되지는 않는다.
4. 즉 검색 품질과 생성 품질은 따로 잡아야 한다.

### 3.2 데이터

1. 공식 원천 데이터는 `openshift-docs` 다.
2. 답변은 반드시 공식 문서에 기대야 한다.
3. 다만 특정 minor 가 아직 미정이므로, 지금은 **validation-mode 중심**으로 품질을 끌어올린다.
4. 나중에 target minor 가 정해지면 source profile 전환으로 release-mode 로 이동한다.

### 3.3 출처

1. citation 은 옵션이 아니라 필수다.
2. citation 은 파일명 나열이 아니라 **실제 클릭 가능한 문서 링크**여야 한다.
3. 출처는 HTML viewer 로 열려야 한다.
4. 이상적인 상태는 “답변이 맞고, 출처도 맞고, 클릭해도 맞는 문서가 뜨는 것”이다.

### 3.4 심사 리스크

지금 가장 큰 리스크는 아래 네 가지다.

1. 기본 정의 질문 대응 부족
2. 한국어 인코딩/문구 깨짐
3. widened corpus 에서 raw retrieval baseline 약함
4. 실런타임과 검증 코퍼스의 불일치 가능성

## 4. 이 프로젝트의 이상적인 챗봇 모습

우리가 바라는 이상적인 챗봇은 다음과 같다.

1. 한국어로 질문해도 자연스럽게 답한다.
2. 기본 개념 질문, 운영 질문, 후속 질문 모두 흔들리지 않는다.
3. “모르겠습니다”를 남발하지 않되, 근거 없이는 과장하지 않는다.
4. 기본 개념 질문에는 안전한 공식 소개 문서로 답한다.
5. 운영 질문에는 관련 디렉터리와 버전을 우선 반영한다.
6. 답변 톤은 짧고 명확하되, 운영 판단에 필요한 핵심 문맥은 놓치지 않는다.
7. 출처는 항상 눈에 보이고 클릭 가능해야 한다.
8. UI 는 화려할 필요 없지만, 보기 불편하거나 깨지면 안 된다.
9. 심사자가 바로 테스트했을 때 “실제로 쓸 수 있겠다”는 느낌을 줘야 한다.

## 5. 하지 말아야 할 것

아래 행동은 금지한다.

1. 문서와 검증 계획만 계속 늘리고, 사용자 눈에 보이는 문제를 뒤로 미루지 않는다.
2. 특정 minor 버전이 미정인데 파이프라인을 특정 minor 에 하드고정하지 않는다.
3. retrieval 품질 문제를 citation UI 로 덮지 않는다.
4. 한글이 깨지는 상태에서 “대체로 된다”고 말하지 않는다.
5. 공식 근거가 없는 일반 상식 답변을 공식 답변처럼 포장하지 않는다.
6. 실행 경로가 불안정한데도 “실행된다”고 넘기지 않는다.

## 8. 앞으로의 작업 순서

이 문서를 기준으로 당장 필요한 작업 순서는 아래와 같다.

### Step A. 사용자 체감 문제 즉시 해결

가장 먼저 잡아야 할 문제:

1. 기본 질문 응답
2. 한글 깨짐
3. UI 문구/정적 파일 깨짐
4. citation click-through 안정성

이 단계의 종료 기준:

- `오픈시프트가 뭐야`
- `OCP가 뭐야`
- `OpenShift가 뭐야`

같은 질문에 한국어 답변이 나와야 한다.

### Step B. 검색 품질 정상화

다음으로 잡아야 할 문제:

1. widened corpus 기준 raw retrieval 약함
2. policy-prepared 의존도가 과도함
3. source_dir/category bias 안정화 필요

이 단계의 종료 기준:

- 기본 운영 질문군에서 기대 문서가 top-k 안에 들어온다.
- citation 이 실제 정답 문서를 가리킨다.

### Step C. 실런타임-실코퍼스 일치성 확보

다음으로 잡아야 할 문제:

1. active manifest 와 실제 OD runtime data 가 같은가
2. smoke 가 진짜 runtime 을 보고 있는가
3. stale runtime data reuse 가 없는가

이 단계의 종료 기준:

- health 에 보이는 코퍼스와 실제 runtime DB 상태가 일치한다.
- “검증은 됐는데 런타임은 다름” 상태가 사라진다.

### Step D. 심사 시나리오 고정

심사에서 바로 나올 질문군을 고정한다.

1. OCP 정의
2. 업그레이드 전 준비사항
3. disconnected 관련
4. operator 관련
5. 네트워크/스토리지/보안 기본 질문
6. 후속 질문 5턴 이상

이 단계의 종료 기준:

- 이 질문군은 시연 전에 전부 pass 여야 한다.

## 9. 오늘의 절대 우선순위

오늘 우선순위는 아래 하나로 정리한다.

**“심사자가 처음 눌렀을 때 바로 보게 되는 문제를 먼저 없앤다.”**

즉 오늘 가장 먼저 보는 항목은:

1. 로컬 UI가 정상 한글로 보이는가
2. 기본 질문이 한국어로 답하는가
3. 출처가 보이고 클릭되는가
4. 스트리밍이 깨지지 않는가

이 네 가지다.

## 10. 맥락이 흔들릴 때 다시 확인할 질문

작업 중 길을 잃었다고 느껴지면 아래 질문으로 복귀한다.

1. 지금 고치는 것이 사용자가 바로 느끼는 문제인가?
2. 지금 고치는 것이 공식 문서 기반 품질을 높이는가?
3. 지금 고치는 것이 심사 리스크를 줄이는가?
4. 지금 고치는 것이 출처 정확성과 click-through 를 개선하는가?
5. 지금 고치는 것이 `localhost:8000` 기준 실제 동작을 더 안정적으로 만드는가?

위 질문 중 세 개 이상에 “예”라고 답하지 못하면 우선순위를 다시 본다.

## 11. 결론

이 프로젝트의 핵심은 문서가 많고 구조가 복잡한 것처럼 보여도 결국 단순하다.

**한국어로 묻고, 공식 문서로 답하고, 출처를 누르면 원문이 열리는 OCP 운영 챗봇을 심사에서 무너지지 않게 만드는 것.**

앞으로 모든 작업은 이 문장을 기준으로 판단한다.

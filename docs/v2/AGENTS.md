# DOCS V2 가이드

## 개요
`docs/v2/`는 브랜치 의도, 계약, 단계 보고서, 검증 완료 상태를 담는 권위 계층이다. 왜 저장소가 이렇게 구성됐는지, 어떤 증거가 권위인지 설명하는 곳이다.

## 어디를 봐야 하나
| 필요 사항 | 위치 | 메모 |
|-----------|------|------|
| 저장소 재진입용 문서 | `repository-map.md`, `project-plan-summary.md` | 가장 빠르게 문맥을 되찾는 경로 |
| 아키텍처와 범위 | `architecture-blueprint.md`, `source-scope.md`, `source-profile-layer.md` | 상위 설계 |
| 평가 권위 문서 | `evaluation-spec.md`, `stage10-evaluation-report.md` | 품질 게이트의 의미 |
| 런타임 경로 권위 문서 | `stage08-live-runtime-quality-report.md`, `stage12-live-runtime-report.md` | 서빙 경로 검증 |
| 갱신/롤백 권위 문서 | `stage07-refresh-activate-rollback-report.md`, `stage11-readiness.md` | 운영 정당성 |
| 실행 경로 권위 문서 | `stage14-runtime-launch.md` | 원커맨드 운영자 런치 검증 |
| 날짜 기준 증거 묶음 | `2026-04-01/` | 스냅샷 보고서이며 유일한 권위는 아님 |

## 로컬 규칙
- 결정/보고 주제마다 문서를 하나씩 두는 편을 선호한다. 파일명은 의도적으로 단계 중심이고 설명적이다.
- 이 문서들은 권위와 증거를 구분한다. 어떤 보고서는 현재의 진실을 선언하고, 어떤 것은 과거 기록이나 보조 자료다.
- 모든 주장에는 파일 증거, 보고서 ID, 스크립트 출력 같은 근거를 밀착시킨다.

## 안티패턴
- 명확한 권위 문서 없이 같은 상태 주장을 여러 문서에 복제하지 않는다.
- 날짜 폴더 보고서를 현재 동작의 유일한 진실 원천으로 만들지 않는다.
- 대응하는 보고서/증거 경로 연결 없이 어떤 단계를 완료로 적지 않는다.

## 작성 스타일
- 직설적이고 운영 중심이며 군더더기가 적은 문체.
- 애매할 수 있으면 “무엇이 권위이고 무엇이 아닌지”를 드러내는 표현을 우선한다.
- 새 단계나 계약을 문서화할 때는 정확한 스크립트, 출력, 통과 기준을 함께 적는다.

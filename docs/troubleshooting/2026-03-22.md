## 1. Red Hat 공식문서 스크래핑 404 오류 (7건)

### 증상
`make scrape` 실행 시 OCP 31개 중 7개 URL이 `404 Client Error: Not Found` 반환.

```
[ERROR] .../monitoring/about-openshift-container-platform-monitoring: 404
[ERROR] .../building_applications/setting-quotas-for-resources: 404
[ERROR] .../nodes/automatically-scaling-pods-with-the-horizontal-pod-autoscaler: 404
[ERROR] .../updating_clusters/understanding-openshift-updates: 404
[ERROR] .../nodes/nodes-pods-autoscaling: 404
[ERROR] .../logging/cluster-logging: 404
[ERROR] .../scalability_and_performance/recommended-etcd-practices: 404
```

추가로 `ocp_route_config.md`는 200 OK지만 272자(사실상 빈 파일)로 저장됨.

### 원인
Red Hat 문서 사이트가 OCP 4.17 버전에서 URL 구조를 대폭 변경함.
- slug 이름 변경 (예: `setting-quotas-for-resources` → `quotas`)
- 가이드 분리/통합 (예: `networking/configuring-routes` → `ingress_and_load_balancing/routes`)
- html-single 전환 (예: `html/monitoring/...` → `html-single/monitoring/index`)

### 해결

`scripts/scrape_docs.py`에서 URL 8개 수정:

| 문서 | 변경 전 (404) | 변경 후 |
|------|-------------|---------|
| monitoring | `html/monitoring/about-openshift-container-platform-monitoring` | `html-single/monitoring/index` |
| resource_quota | `html/building_applications/setting-quotas-for-resources` | `html/building_applications/quotas` |
| autoscaling | `html/nodes/automatically-scaling-pods-with-the-horizontal-pod-autoscaler` | `html/nodes/automatically-scaling-pods-with-the-custom-metrics-autoscaler-operator` |
| updating | `html/updating_clusters/understanding-openshift-updates` | `html/updating_clusters/understanding-openshift-updates-1` |
| pod_autoscaling | `html/nodes/nodes-pods-autoscaling` | `html-single/nodes/index` |
| logging | `html/logging/cluster-logging` | `html/logging/logging-6-0` |
| route_config | `html/networking/configuring-routes` | `html/ingress_and_load_balancing/routes` |
| etcd | `html/scalability_and_performance/recommended-etcd-practices` | `html/scalability_and_performance/recommended-performance-and-scalability-practices-2` |

수정 후 `rm data/raw/ocp_route_config.md` (272자 빈 파일 삭제) → `make scrape` 재실행.

### 교훈
- Red Hat 문서는 마이너 버전마다 URL slug가 바뀔 수 있음
- 스크래퍼에 "파일이 존재하면 SKIP" 로직이 있어서, 빈 파일(272자)도 SKIP됨 → 수동 삭제 필요
- 향후 개선: 파일 크기가 일정 기준 이하면 재다운로드하는 로직 추가 검토

## 3. 마크다운 테이블 렌더링 안 됨

### 증상
챗봇이 표 형태(`| 항목 | 설명 |`)로 응답하면 마크다운 원문이 그대로 텍스트로 출력됨.

### 원인
`frontend/index.html`의 커스텀 마크다운 렌더러(`renderMarkdown()`)에 테이블 파싱 로직이 없었음. 볼드, 코드블록, 리스트만 지원.

### 해결
- `renderMarkdownTable()` 함수 추가: `|---|---|` 구분선 패턴 감지 → HTML `<table>` 변환
- `renderMarkdown()`에서 테이블을 먼저 추출한 후 나머지 규칙 적용
- `.message table`, `.message th`, `.message td` CSS 스타일 추가

## 4. 검색 결과에 같은 문서가 3개 이상 등장

### 증상
"Pod 리소스 제한 설정" 검색 시 5개 결과 중 3개가 `k8s_task_memory_resource.md`에서 나옴. 출처 다양성 부족.

### 원인
Hybrid Reranking (Semantic 70% + BM25 30%) 후 상위 결과를 그대로 반환. 같은 문서의 여러 청크가 높은 점수를 받으면 결과를 독점.

### 해결
`src/retriever/__init__.py`에 소스별 중복 제거 추가:
```python
max_per_source = 2
source_count: dict[str, int] = {}
for r in ranked:
    src = r.metadata.get("source", "")
    if source_count.get(src, 0) >= max_per_source:
        continue
    top_results.append(r)
    source_count[src] = source_count.get(src, 0) + 1
```

### 교훈
- MMR(Maximal Marginal Relevance) 같은 복잡한 알고리즘 대신 단순 카운트 제한이 효과적
- 심플 이즈 더 베스트

## 5. "세줄요약" vs "네줄요약" 캐시 오탐

### 증상
"OCP Pod 세줄요약" 질문 후 "OCP Pod 네줄요약"을 물으면 세줄요약 캐시 응답이 그대로 반환됨.

### 원인
두 쿼리의 코사인 유사도가 0.95+ → `SemanticCache`가 같은 질문으로 판단. 숫자("3" vs "4")만 다른데 임베딩에서는 거의 동일하게 인코딩됨.

### 해결
`src/cache/__init__.py`에 숫자 비교 로직 추가:
```python
@classmethod
def _extract_numbers(cls, text: str) -> list[str]:
    return cls._NUMBER_RE.findall(text)

# _is_cache_compatible()에서 숫자가 다르면 캐시 미스
if cls._extract_numbers(query) != cls._extract_numbers(cached_query):
    return False
```

### 교훈
- 임베딩 기반 유사도만으로는 세밀한 차이를 구분 못함
- 코사인 유사도 + 토큰 가드 + 숫자 가드 = 3중 검증이 안전

## 6. Query Rewrite가 동사를 바꿔버림

### 증상
대화 이력: "OCP에서 Pod 생성 방법"
새 질문: "그거 삭제하려면?"
리라이트 결과: "OCP에서 Pod 생성 방법을 더 자세히..." (삭제 → 생성으로 바뀜)

### 원인
9B 양자화 모델(Qwen3.5-9B-4bit)이 장황한 프롬프트의 규칙을 제대로 따르지 못함. 대화 이력의 "생성" 키워드에 끌려가서 사용자의 "삭제" 의도를 무시.

### 해결
`src/session/__init__.py`의 `REWRITE_PROMPT`를 few-shot 예시 기반으로 전면 개편:
```
예시:
- 이력: "OCP에서 Pod 생성 방법" → 최신: "그거 삭제하려면?" → 출력: "OCP에서 Pod를 삭제하려면?"

규칙:
- 대명사(그거, 이거, 그것)를 이력의 구체적 대상으로 교체
- 최신 질문의 동사(삭제, 수정, 조회 등)는 절대 바꾸지 마라
- 재작성된 질문만 출력 (설명 없이)
```

### 교훈
- 소형 LLM에는 장황한 규칙보다 **구체적 few-shot 예시**가 훨씬 효과적
- 규칙은 3줄 이내로 핵심만

## 7. "네임스페이스가 뭐야?" 검색 결과 부실

### 증상
"네임스페이스가 뭐야?" 질문에 관련 없는 문서가 출처로 표시됨. 답변 품질 매우 낮음.

### 원인
코퍼스에 Namespace/Project 전용 문서가 없음. 전체 75개 파일 중 "namespace"는 다른 문서에서 4줄 언급 수준.

### 해결
검색/리라이트 문제가 아닌 **코퍼스 부재** 문제로 진단 → 코퍼스 보강 계획 수립:
- `scripts/scrape_docs.py`에 OCP Projects 문서 URL 추가
- `scripts/generate_docs.py`에 `ocp-namespace-project-ko.md` 한국어 합성 토픽 추가

### 교훈
- RAG 시스템은 코퍼스가 답변 품질의 상한선
- 검색/리랭킹 아무리 튜닝해도 문서가 없으면 답을 못함
- "답변이 이상하다" → 먼저 코퍼스에 해당 주제 문서가 있는지 확인

## 1. Windows 콘솔 UnicodeEncodeError (cp949)

### 증상
인덱스 빌드, eval 스크립트 등 한국어 출력 시 `UnicodeEncodeError: 'cp949' codec can't encode character '\u2014'` 에러 발생.

### 원인
Windows 명령 프롬프트의 기본 인코딩이 `cp949`(한국어 Windows). Python `print()`가 em-dash(`—`), 특수 유니코드 문자를 cp949로 변환 시 실패.

Mac에서는 기본 UTF-8이라 문제 없었음.

### 해결
두 가지 방법:

**방법 1: 환경변수 설정**
```batch
set PYTHONIOENCODING=utf-8
python scripts/build_index.py
```

**방법 2: 소스 코드에서 특수문자 제거**
```python
# Before
print("검색 정확도 — 80.6%")
# After
print("검색 정확도: 80.6%")
```

### 교훈
- Mac과 Windows의 기본 콘솔 인코딩이 다름 (UTF-8 vs cp949)
- 크로스 플랫폼 프로젝트에서는 print문에 ASCII 범위 외 특수문자 사용 자제
- 배치 스크립트에 `set PYTHONIOENCODING=utf-8` 기본 포함 권장

## 3. 서버 재시작해도 인덱스가 갱신되지 않음 (반복 발생)

### 증상
- `index.bat`으로 인덱스 재빌드 (14,373 벡터)
- `run.bat`으로 서버 실행 (또는 `--reload` 모드)
- `/api/stats`에서 여전히 이전 벡터 수(14,215 또는 14,301) 표시
- 새로 추가한 문서에 대한 검색이 안 됨

### 원인
`IVFIndex`는 `@app.on_event("startup")`에서 **한 번만** 디스크에서 메모리로 로드됨.

```python
@app.on_event("startup")
async def startup():
    global pipeline
    index = IVFIndex.load(INDEX_DIR)  # 여기서 1회 로드
```

uvicorn `--reload` 모드는 **코드 파일 변경**을 감지해서 재시작하지만:
- `data/index/*.npy` 파일 변경은 감지하지 않음
- 코드가 안 바뀌면 재시작 자체가 안 됨
- 재시작 되더라도 이전 Python 프로세스가 포트를 놓지 않는 경우 있음

### 해결
**인덱스 재빌드 후에는 반드시 풀 재시작:**
```batch
kill.bat
run.bat
```

`/api/stats` 응답으로 확인:
```json
{"index": {"total_vectors": 14373, "n_probe": 16}}
```

### 교훈
- `--reload`는 코드 개발용이지, 데이터 갱신용이 아님
- 인덱스 재빌드 → 반드시 kill → run 순서
- `/api/stats`로 벡터 수 확인하는 습관 필수
- **이 문제로 오늘만 3번 이상 삽질함** — 인덱스 재빌드 후 서버가 자동으로 새 인덱스를 로드하는 기능 추가 검토

## 4. 시맨틱 캐시가 "못 찾겠다" 응답을 반환

### 증상
- 코퍼스에 CrashLoopBackOff 문서 추가 후 질문
- "제공된 문서에서 해당 정보를 찾지 못했습니다" 응답이 계속 나옴
- 새로 추가한 문서가 검색되지 않는 것처럼 보임

### 원인
시맨틱 캐시가 **데이터 추가 전**에 저장한 응답을 반환.

흐름:
1. 문서 추가 전: "CrashLoopBackOff 해결방법?" → 검색 실패 → "찾지 못했습니다" 응답 → **캐시 저장**
2. 문서 추가 후 인덱스 재빌드
3. 같은 질문 → 캐시 유사도 0.95+ → 캐시 히트 → **옛날 "못 찾겠다" 응답 반환**
4. 실제 검색은 수행되지 않음

시맨틱 캐시는 인메모리(`dict`)이므로 서버 재시작 시 초기화되지만, 3번 문제(서버가 제대로 재시작 안 됨)와 겹치면서 캐시가 계속 살아있는 것처럼 보였음.

### 해결
- 서버 풀 재시작으로 캐시 초기화
- 또는 `/api/cache/clear` API 호출

### 근본 대책 (미구현)
```python
# pipeline.py에서 에러성 응답은 캐시하지 않는 로직 강화
FAIL_KEYWORDS = ["찾지 못했습니다", "해당 정보를 찾", "데이터가 부족"]
if any(kw in answer for kw in FAIL_KEYWORDS):
    # 캐시 저장 스킵
    pass
```

### 교훈
- "데이터가 없다"는 응답을 캐시하면 데이터 추가 후에도 계속 같은 응답이 나옴
- 실패 응답 캐시 방지 로직 필수

## 5. IVF n_probe config 변경이 반영되지 않음

### 증상
`src/config.py`에서 `IVF_N_PROBE`를 6→10으로 변경했는데, `/api/stats`에서 여전히 `n_probe: 6` 표시.

### 원인
`IVFIndex.load()`가 `index_meta.json`에 저장된 빌드 시점의 n_probe 값을 사용:

```python
# Before (버그)
index = cls(
    dim=meta["dim"],
    n_clusters=meta["n_clusters"],
    n_probe=meta["n_probe"],  # ← 빌드 시점 값 (6)
)
```

`index_meta.json`에는 인덱스를 처음 빌드할 때의 n_probe(6)가 저장되어 있음. config에서 아무리 바꿔도 load 시 meta 값으로 덮어써짐.

### 해결
```python
# After (수정)
from src.config import IVF_N_PROBE

index = cls(
    dim=meta["dim"],
    n_clusters=meta["n_clusters"],
    n_probe=IVF_N_PROBE,  # ← config 값 우선
)
```

### 교훈
- 설정값이 "저장된 값"과 "런타임 값" 두 곳에 있으면 우선순위를 명확히 해야 함
- 인덱스 메타데이터는 구조 정보(dim, n_clusters)만 저장하고, 검색 파라미터(n_probe)는 런타임 config 우선
- 버그가 은밀해서 찾기 어려움 — `/api/stats`로 실제 값 확인하는 습관이 잡아냈음

## 6. IVF 클러스터 누락으로 문서 검색 실패

### 증상
CrashLoopBackOff 트러블슈팅 문서가 인덱스에 있는데도(12개 청크) 벡터 검색에서 안 잡힘.

### 원인
IVF 인덱스 구조의 한계:
- 14,373개 벡터를 16개 클러스터로 분할
- `n_probe=6`이면 검색 시 6개 클러스터만 탐색
- CrashLoopBackOff 문서가 쿼리 벡터와 가까운 클러스터가 아닌 다른 클러스터에 배치된 경우 누락

### 해결
`IVF_N_PROBE`: 6 → 16 (전체 클러스터 탐색)

```python
# src/config.py
IVF_N_PROBE = int(os.getenv("IVF_N_PROBE", "16"))
```

### 성능 영향
- n_probe=6: ~10ms
- n_probe=16: ~26ms
- 14,373 벡터 규모에서 16ms 차이는 무시 가능

### 교훈
- 벡터 수가 적으면 IVF 근사검색의 이점이 크지 않음 — 전수탐색이 더 안전
- 수만 벡터 이하에서는 n_probe를 n_clusters와 동일하게 설정해도 속도 영향 미미
- FAISS 같은 라이브러리도 동일한 trade-off 존재

## 7. 하이브리드 리랭킹이 한국어 문서를 밀어냄 (핵심 이슈)

### 증상
- `scripts/eval_questions.py` 평가에서 CrashLoopBackOff 검색 점수 0.732 (PASS)
- **하지만 실제 챗봇에서** "데이터가 부족합니다" 응답
- 서버 재시작, n_probe=16, 14,373 벡터 확인 후에도 동일 증상
- eval과 실제 챗봇 결과가 불일치

### 원인 분석

**eval 스크립트 vs 실제 파이프라인 차이:**
- eval: `retriever.retrieve()` → 하이브리드 검색 → 점수 반환 (여기까지만)
- 파이프라인: `retriever.retrieve()` → 하이브리드 검색 → `build_context()` → LLM 전달 → **LLM이 판단**

eval 점수 0.732는 **가장 높은 문서**의 점수. 하지만 그 문서가 `ocp-core-concepts-ko.md`(일반 개념)이지, `ocp-crashloop-troubleshooting-ko.md`(실제 해결방법)이 아니었음.

**직접 벡터 검색 결과:**
```
1. score=0.640 src=ocp-crashloop-troubleshooting-ko.md  ← 타겟 문서 1위!
2. score=0.597 src=ocp-core-concepts-ko.md
```

**하이브리드 리랭킹 후:**
```
1. score=0.732 src=ocp-core-concepts-ko.md      (sem=0.743, kw=0.715)
2. score=0.699 src=ocp_troubleshooting_os.md     (sem=0.636, kw=0.794)  ← 영문
3. score=0.699 src=ocp_troubleshooting_pods.md   (sem=0.636, kw=0.794)  ← 영문
4. score=0.697 src=k8s_pod_disruptions.md        (sem=0.626, kw=0.802)  ← 영문
5. score=0.643 src=ocp_troubleshooting_os.md     (sem=0.595, kw=0.715)  ← 영문 (같은 소스 2번째)
>>> ocp-crashloop-troubleshooting-ko.md는 TOP 5에서 탈락!
```

**원인 체인:**
1. **BM25가 영문 문서에 과도한 점수**: "CrashLoopBackOff"는 영어 단어 → 영문 문서에서 정확히 토큰 매칭 → BM25 점수 0.79~0.80
2. **한국어 문서의 BM25 불리**: 한국어 텍스트는 공백 기반 BM25 토크나이저에서 불리 (교착어 특성)
3. **`max_per_source=2`**: `ocp_troubleshooting_os.md`에서 2개, `ocp_troubleshooting_pods.md`에서 2개 → 5개 슬롯 중 4개를 영문 문서가 차지
4. **한국어 해결방법 문서 탈락**: semantic score는 높지만 BM25에서 밀려서 최종 순위 하락
5. **LLM이 받는 컨텍스트**: 영문 에러 테이블 + 일반 개념 설명 → "해결방법에 대한 데이터가 부족하다"

### 해결 (4건)

**1. `max_per_source` 축소 (2 → 1)**
```python
# src/retriever/__init__.py
max_per_source = 1  # 같은 문서에서 1개만 → 다양한 문서 확보
```

**2. 한국어 쿼리 시 한국어 문서 부스트**
```python
# 한국어 감지
has_korean = bool(re.search(r'[\uac00-\ud7af]', query))

# 한국어 쿼리 + 한국어 문서 → +0.08 부스트
if has_korean and source.endswith("-ko.md"):
    combined_score += 0.08
```

**3. 컨텍스트 크기 확대 (4000 → 6000자)**
```python
def build_context(self, results, max_chars=6000):
```
5위 문서도 컨텍스트에 포함되도록 여유 확보.

**4. 시스템 프롬프트 완화**
```
# Before
"참고 문서에 없는 내용은 절대 답변하지 마세요."

# After
"참고 문서에 관련 내용이 조금이라도 있으면, 그 내용을 기반으로 최선의 답변을 제공하세요."
```

### 수정 후 결과
```
1. score=0.812 src=ocp-core-concepts-ko.md         [KO]
2. score=0.699 src=ocp_troubleshooting_os.md
3. score=0.699 src=ocp_troubleshooting_pods.md
4. score=0.697 src=k8s_pod_disruptions.md
5. score=0.696 src=ocp-crashloop-troubleshooting-ko.md [KO] ← 진입 성공!
```

Context: 6003자, CrashLoopBackOff 해결방법 내용 포함 확인. 실제 챗봇에서 정상 답변 출력 확인.

### 교훈

1. **eval 점수 ≠ 실제 답변 품질**: eval은 "가장 높은 점수"만 보지만, 실제로는 "LLM에 전달되는 컨텍스트 전체"가 중요
2. **BM25의 언어 편향**: 공백 기반 토크나이저는 영어에 유리하고 한국어에 불리. 다국어 RAG에서는 보정 필수
3. **"검색이 잘 되는데 답변이 안 된다"는 두 가지 가능성**:
   - 검색 결과의 상위 문서가 "질문과 관련은 있지만 답변에 필요한 내용이 아닌" 경우
   - LLM이 컨텍스트를 보고 "부족하다"고 과도하게 거부하는 경우
4. **근본 원인 추적이 중요**: 서버 재시작, n_probe 변경, 인덱스 재빌드를 반복해도 안 되면, 파이프라인 각 단계의 실제 데이터를 직접 확인해야 함

## 8. kill.bat 실행해도 Python 프로세스가 안 죽음

### 증상
```batch
kill.bat
```
실행 후에도 `http://localhost:8000` 접속 가능. `tasklist`에 python.exe 프로세스 남아있음.

### 원인
Claude Code의 백그라운드 에이전트가 별도 Python 프로세스를 실행 중이었음. kill.bat은 특정 PID만 종료하거나 `taskkill /IM python.exe`로 전체 종료하는데, 백그라운드 프로세스가 포트를 계속 점유.

### 해결
```batch
taskkill /F /PID <특정_PID>
```
또는 모든 Python 프로세스 강제 종료:
```batch
taskkill /F /IM python.exe
```

### 교훈
- Claude Code가 백그라운드로 실행한 프로세스는 대화가 끝나도 살아있을 수 있음
- kill.bat에 포트 기반 종료 로직 추가 검토:
  ```batch
  for /f "tokens=5" %%a in ('netstat -ano ^| findstr :8000') do taskkill /F /PID %%a
  ```

## 9. Red Hat 문서 스크래핑 — 빈 파일 (6건, <1KB)

### 증상
`data/raw/` 내 6개 파일이 1KB 미만. 실제 내용은 HTML 껍데기만 있고 본문 없음.

### 원인
Red Hat 공식 문서 사이트가 JavaScript SPA(Single Page Application)로 구현됨.
- `requests` + `BeautifulSoup` 조합은 서버에서 내려온 초기 HTML만 파싱
- JavaScript로 렌더링되는 본문 컨텐츠는 파싱 불가
- `<div id="root"></div>` 같은 빈 컨테이너만 남음

### 해결 (미착수)
Playwright 또는 Selenium으로 JS 렌더링 후 파싱:
```python
from playwright.async_api import async_playwright

async with async_playwright() as p:
    browser = await p.chromium.launch()
    page = await browser.new_page()
    await page.goto(url)
    await page.wait_for_selector('.article-body')  # JS 렌더링 대기
    content = await page.content()
```

### 교훈
- 현대 웹사이트의 상당수가 SPA → `requests`만으로는 한계
- 스크래핑 후 파일 크기 검증 로직 필요 (1KB 미만이면 경고)
- 대안: Red Hat에서 제공하는 raw AsciiDoc 소스 활용 검토

## 10. `data/raw/` 디렉토리 Windows에 없음

### 증상
`scripts/generate_docs.py` 실행 시 `FileNotFoundError: data/raw/` 경로를 찾을 수 없음.

### 원인
`data/raw/`가 `.gitignore`에 포함되어 있어서 git으로 Mac → Windows 전송 시 디렉토리 자체가 누락됨.

### 해결
```batch
mkdir data\raw
```
이후 `generate_docs.py`로 합성 문서 재생성.

### 교훈
- `.gitignore`에 포함된 디렉토리는 새 환경에서 수동 생성 필요
- README나 셋업 스크립트에 `mkdir -p data/raw` 포함 권장
- `scripts/generate_docs.py`에 `os.makedirs(output_dir, exist_ok=True)` 추가 검토

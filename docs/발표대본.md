## 2. 왜 직접 만들었나 (~30초)

이번 과제에서 LangChain이나 LlamaIndex 같은 외부 RAG 프레임워크 사용이 금지되어 있었습니다.

그래서 벡터 검색 인덱스부터 키워드 검색, 리랭킹, 캐시까지 전부 직접 구현했습니다. 벡터 인덱스는 numpy로 IVF 구조를 만들었고, FAISS를 안 쓴 이유는 14,000개 벡터 규모에서는 과도하기도 하고, 클러스터 수나 탐색 범위 같은 하이퍼파라미터를 직접 제어하고 싶었기 때문입니다.

【슬라이드: 기술 스택】

## 3. 파이프라인 흐름 (~90초)

전체 파이프라인은 10단계인데, 크게 세 덩어리로 나눌 수 있습니다.

【슬라이드: 파이프라인 전체도】

**첫 번째는 질문 전처리입니다.**

사용자가 "그거 삭제하려면?" 이런 식으로 대명사를 쓰면 검색이 안 됩니다. 그래서 쿼리 리라이팅 단계에서 대화 이력을 보고 "OCP Pod 삭제하려면?"으로 변환합니다. 단순 패턴은 정규식으로 빠르게 처리하고, 맥락이 필요한 건 LLM에 Few-shot 예시를 줘서 처리합니다.

그 다음 시맨틱 캐시를 확인합니다. 이전에 거의 같은 질문이 있었으면 LLM을 다시 호출하지 않고 바로 답을 줍니다. 임계값은 코사인 유사도 0.95이고, 숫자나 엔티티가 다른 경우를 잡기 위해 3중 검증을 추가했습니다.

【슬라이드: 검색 구조】

**두 번째는 검색입니다.**

벡터 검색과 BM25 키워드 검색을 동시에 돌립니다. 벡터 검색은 의미가 비슷한 문서를, BM25는 "CrashLoopBackOff" 같은 정확한 키워드를 잡습니다.

두 결과를 합쳐서 리랭킹하는데, 여기서 핵심은 쿼리 유형별로 가중치를 다르게 준다는 겁니다. 트러블슈팅 질문이면 키워드 비중을 올리고, 개념 질문이면 벡터 비중을 올립니다. 고정 가중치로는 두 유형을 동시에 최적화할 수 없기 때문입니다.

그리고 선택된 청크의 앞뒤 청크도 함께 가져옵니다. 512자 단위로 잘랐기 때문에 증상은 청크 0, 해결방법은 청크 2에 있는 경우가 많거든요.

【슬라이드: LLM 응답】

**세 번째는 응답 생성입니다.**

검색된 문서를 컨텍스트로 넣고 Qwen 3.5 9B 모델로 답변을 생성합니다. SSE 스트리밍으로 토큰 단위로 실시간 전송하고요. 운영 모드와 학습 모드 두 가지를 지원해서, 같은 질문이라도 운영자한테는 명령어 위주로, 학습자한테는 설명 위주로 답변합니다.

## 4. 설계 결정 하이라이트 (~60초)

【슬라이드: 설계 결정】

몇 가지 설계 결정을 짚어드리겠습니다.

**임베딩 모델 선택.** 처음에 영어 전용 MiniLM을 썼는데, "Pod"와 "파드"의 유사도가 0.42밖에 안 나왔습니다. 다국어 모델로 바꾸니까 0.73으로 올라갔습니다. 한영 혼재 문서에서는 다국어 임베딩이 필수라는 걸 실험으로 확인했습니다.

**BM25 동의어 사전.** 91쌍의 한영 동의어를 수작업으로 만들었습니다. "파드 에러"로 검색해도 영문 문서의 "pod error"가 매칭됩니다.

**시맨틱 캐시 3중 검증.** 유사도 0.95만으로는 "세줄요약"과 "네줄요약"을 구분 못 합니다. 유사도가 0.97인데 답변은 달라야 하거든요. 그래서 숫자 비교, 엔티티 비교, 토큰 교집합까지 추가 검증합니다.

# Playbook Wiki Pipeline Plain Guide

이 문서는 `기술을 잘 모르는 사람도 지금 시스템이 어떻게 돌아가는지 이해할 수 있게` 쓰는 설명서다.

이 문서는 `active contract`가 아니라 `이해를 돕는 reference 문서`다.

## 1. 이 제품은 한 줄로 뭐 하는 물건인가

이 제품은 문서를 그냥 저장하는 툴이 아니다.

이 제품은:

1. 여러 기술 문서를 모아서
2. 읽기 좋은 위키 문서로 다시 만들고
3. 그 위키를 챗봇이 참고해서 답하게 만드는 제품이다.

즉:

`문서 수집 공장 + 위키 제작 공장 + 챗봇 근거 검색기`

이 세 개가 합쳐진 제품이다.

---

## 2. 가장 중요한 구분

이 시스템은 크게 두 부분으로 나뉜다.

### A. 문서를 만드는 파이프라인

원문을 가져와서 위키 문서로 다시 만드는 과정이다.

### B. 챗봇이 문서를 찾는 파이프라인

사용자가 질문했을 때, 어떤 문서를 근거로 쓸지 찾는 과정이다.

이 둘을 섞어서 생각하면 계속 헷갈린다.

---

## 3. 최종 문서는 어떤 형태로 저장되나

짧게 말하면:

- 저장은 `runtime markdown/book`
- 출력은 `HTML viewer`

이다.

### 3-1. 저장 형식

현재 active 위키 본문은 주로 아래 경로에 저장된다.

- `data/wiki_runtime_books/full_rebuild/*.md`

예:

- `data/wiki_runtime_books/full_rebuild/support.md`

이 파일 확장자는 `.md` 이다.

하지만 이걸 보고 `그냥 옛날 markdown 문서구나` 라고 보면 안 된다.

이 `.md`는:

- 섹션 구조
- 코드 블록
- 그림 참조
- 문서 관계

를 전제로 만들어진 `runtime markdown` 이다.

즉:

`최종 제품의 본체 = 단순 md 파일 1개`

가 아니라

`runtime markdown + relation + figure asset + overlay + manifest`

묶음이다.

---

## 4. 사용자는 뭘 보나

사용자는 `.md` 파일을 직접 보지 않는다.

사용자는 `HTML viewer` 를 본다.

예를 들면 이런 경로다.

- `/playbooks/wiki-runtime/active/<slug>/index.html`
- `/wiki/figures/<book>/<asset>/index.html`

즉:

- 저장은 markdown/runtime 구조
- 표시는 HTML viewer

로 분리되어 있다.

왜 이렇게 하냐면, HTML viewer가 있어야 아래를 제대로 처리할 수 있기 때문이다.

1. 이미지 표시
2. 그림 크기 조절
3. 코드 복사 버튼
4. 섹션 앵커 이동
5. 위키 링크 카드
6. 사용자 오버레이

그래서 `viewer HTML`은 단순 장식이 아니라 제품의 실제 읽기 면이다.

---

## 5. 챗봇은 뭘 참고하나

챗봇은 사용자가 보는 HTML 화면을 직접 읽는 게 아니다.

챗봇은 검색용으로 따로 정리된 문서 조각을 참고한다.

쉽게 말하면:

1. 문서를 작은 조각(chunk)으로 나눈다.
2. 그 조각들을 검색 인덱스에 넣는다.
3. 질문이 들어오면 그 조각 중 관련 있는 것만 뽑는다.
4. 그 조각을 근거로 답변을 만든다.

즉 챗봇은 아래를 참고한다.

- normalized docs
- bm25 corpus
- vector index
- relation graph
- selected citations

사용자가 보는 위키와 `내용 기준선`은 같아야 맞다.

하지만 형식은 다르다.

- 사용자: HTML viewer
- 챗봇: 검색용 chunk + citation context

---

## 6. OCR은 대체 뭐냐

OCR은 아주 쉽게 말하면:

`이미지 안 글자를 읽어서 텍스트로 바꾸는 기술`

이다.

예를 들면:

- 스캔 PDF
- 이미지로만 된 문서
- 글자가 사진처럼 박혀 있는 문서

이런 것에서 글자를 복원할 때 쓴다.

### 중요한 점

OCR은 `그림을 버리고 글자만 남기기 위한 기술`이 아니다.

OCR의 역할은:

- 텍스트를 복원하는 것

이다.

그리고 그림/도해는:

- 별도 figure asset 으로 보존해야 한다.

즉 좋은 파이프라인은 반드시 둘 다 해야 한다.

1. OCR로 글자를 살린다.
2. 그림 자산도 같이 살린다.

만약 OCR만 하고 그림을 버리면:

- 문단이 깨지고
- 그림 설명이 사라지고
- 백과사전 품질이 떨어진다.

사용자가 계속 지적한 우려가 바로 이 부분이다.

그 우려는 맞다.

---

## 7. 지금 OCR은 어떤 식으로 작동하나

현재 설명상 customer/scan lane은 아래 순서로 fallback 을 쓴다.

1. `docling`
2. `mdls`
3. `string_scan`
4. `rendered_ocr`

이 뜻은:

`OCR 하나만 쓰는 게 아니라, 텍스트를 최대한 복원할 수 있는 여러 경로를 순서대로 시도한다`

는 뜻이다.

### 중요한 오해 정리

OCR이 작동하는 조건은:

- `이미지에 글씨가 없을 때`

가 아니다.

정확히는:

- 원문에서 텍스트를 구조적으로 충분히 못 뽑을 때
- 텍스트 품질이 낮을 때
- 스캔 PDF처럼 OCR 없이는 본문이 안 나올 때

fallback 으로 들어가야 맞다.

즉:

`OCR = 구조 텍스트 복원용 후순위 안전장치`

라고 보면 된다.

---

## 8. 폐쇄망에서 이미지가 그때그때 인터넷에서 로딩되나

제품 기준으로는 그러면 안 된다.

정답은:

`수집 시점에 이미지를 로컬 자산으로 가져와서, viewer 는 그 로컬 자산만 봐야 한다`

이다.

현재 구조도 그 방향이다.

figure 는 별도 viewer path 를 가진다.

예:

- `/wiki/figures/.../index.html`

이 말은:

- viewer 단계에서 외부 사이트 이미지를 실시간으로 읽는 게 아니라
- 이미 수집된 figure asset 을 보여줘야 한다

는 뜻이다.

원문 URL은 provenance 용도로 남을 수 있다.

하지만 제품의 실제 표시 면은 로컬 자산 기반이어야 한다.

폐쇄망 용도라면 이 원칙은 더 강하게 지켜져야 한다.

---

## 9. 최종 파이프라인은 정확히 몇 단계인가

지금 기준으로는 6단계로 이해하는 게 가장 쉽고 정확하다.

---

## 10. 1단계: 원천 수집

여기서는 문서 원문을 가져온다.

가져오는 대상:

- 공식 repo AsciiDoc
- published HTML single fallback
- customer 업로드 문서

왜 여러 개를 쓰나:

- 공식 repo 가 가장 좋지만
- 모든 책이 항상 완벽하게 repo만으로 닫히지 않기 때문이다.

그래서 원칙은:

`repo first, fallback only when needed`

이다.

---

## 11. 2단계: 파싱 / 정규화

여기서는 원문을 기계가 읽기 좋은 공통 구조로 바꾼다.

예:

- HTML 태그 정리
- heading 추출
- code block 추출
- 표/문단 분리

여기서 쓰는 도구 예시:

- `BeautifulSoup`
- `MarkItDown`
- OCR fallback tools

이 단계는 아직 “위키”가 아니다.

그냥 `원문을 정리된 재료`로 만드는 단계다.

---

## 12. 3단계: structured book 만들기

여기서는 정리된 재료를 `책 구조`로 다시 묶는다.

즉:

- 제목
- 장/절
- 문단
- 코드
- 표
- 그림 참조

를 가진 `book`을 만든다.

이 단계가 중요한 이유:

우리 제품의 기본 단위는 raw 파일이 아니라 `book`이기 때문이다.

---

## 13. 4단계: reader-grade shaping + figure runtime

여기서는 책을 사람이 읽기 좋게 다듬는다.

예:

- 너무 긴 문단 자르기
- 코드와 설명 분리
- 그림 자산 연결
- figure size 정책 적용

이 단계에서 품질이 낮으면:

- 개행이 이상해지고
- 코드박스가 더러워지고
- 그림이 너무 크거나 작아지고
- 전반적인 위키 품질이 무너진다.

사용자가 계속 지적한 문제 대부분이 여기에 걸려 있다.

---

## 14. 5단계: relation / 검색 인덱스 생성

여기서는 책을 찾을 수 있게 만들고, 책들 사이를 연결한다.

여기서 하는 일:

- entity hub 만들기
- related section 만들기
- figure relation 만들기
- 검색용 인덱스 만들기

이 단계에서 챗봇용 검색도 준비된다.

---

## 15. 6단계: active 승격 + 제품 연결

여기서 가장 중요하다.

좋은 산출물 중에서도 `최종 통과한 것만 active`로 올려야 한다.

이 active만:

- 위키 viewer
- studio
- control tower
- chatbot

이 쓰게 만들어야 한다.

지금 계속 문제가 반복된 이유도 대부분 이 단계가 완전히 잠기지 않았기 때문이다.

즉:

- 옛 데이터가 섞이고
- 비슷한 문서가 같이 뜨고
- 추천질문이 엉뚱해지고
- 영어 문서가 나오고
- 챗봇 품질 평가가 왜곡됐다.

---

## 16. BM25는 뭐냐

BM25는 아주 쉽게 말하면:

`질문에 들어간 단어가 실제 문서에 얼마나 비슷하게 들어 있는지 찾는 검색`

이다.

예:

- 질문에 `etcd`
- 문서에도 `etcd`

같은 단어가 있으면 잘 잡는다.

장점:

- 단어가 정확히 일치하면 강하다
- 기술 문서 검색에 기본기가 좋다

단점:

- 같은 뜻인데 다른 표현이면 약하다

---

## 17. vector search는 뭐냐

vector search는 아주 쉽게 말하면:

`단어가 똑같지 않아도 뜻이 비슷하면 찾는 검색`

이다.

예:

- 질문: “로그 확인 방법”
- 문서: “journalctl로 상태 점검”

단어는 조금 달라도 비슷한 의미로 잡아낼 수 있다.

장점:

- 표현이 달라도 의미 유사성으로 찾을 수 있다

단점:

- 애매한 비슷함도 잡아서 엉뚱한 문서를 끌어올 수 있다

---

## 18. BM25와 vector는 어디서 쓰나

아주 중요하다.

BM25와 vector는:

`문서를 만드는 단계`

에서 쓰는 게 아니다.

이 둘은:

`챗봇이 질문을 받았을 때, 어떤 문서를 참고할지 찾는 검색 단계`

에서 쓴다.

즉:

- 정규화 파이프라인 ≠ BM25/vector
- 검색 파이프라인 = BM25/vector

이다.

---

## 19. GraphDB는 어디서 쓰나

GraphDB도 쓰고 있다.

다만 역할이 다르다.

GraphDB는:

`검색 후보가 나온 뒤, 그 후보 주변의 관계를 확장하는 단계`

에서 쓴다.

쉽게 말하면:

1. BM25가 단어로 찾고
2. vector가 의미로 찾고
3. GraphDB가 주변 연결을 찾아 주고
4. reranker가 최종 순서를 다시 정한다

이다.

즉 GraphDB는:

- 문서 생성기 아님
- 검색 1차 엔진도 아님
- 검색 후 관계 확장 엔진

이다.

현재 실측 trace에 `graph_expand`, `adapter_mode = neo4j`가 뜨는 이유가 이거다.

---

## 20. reranker는 뭐냐

reranker는 마지막 심사관이다.

BM25, vector, GraphDB가 “이것도 후보, 저것도 후보” 하고 많이 가져오면

reranker가:

`그래서 진짜 이 질문에 제일 가까운 근거가 뭐냐`

를 다시 정한다.

쉽게 말하면:

- BM25/vector = 후보 찾기
- GraphDB = 연결 보강
- reranker = 최종 줄세우기

---

## 21. 질문 하나가 들어오면 실제로 무슨 일이 벌어지나

예를 들어 사용자가 질문한다.

`etcd 백업은 어떻게 해?`

그러면 시스템은 대충 이렇게 움직인다.

1. 질문을 정리한다.
2. BM25로 단어 비슷한 문서를 찾는다.
3. vector로 의미 비슷한 문서를 찾는다.
4. 둘을 합친다.
5. GraphDB로 관련된 주변 문서를 본다.
6. reranker로 최종 후보를 다시 정한다.
7. citation 후보를 뽑는다.
8. 근거가 충분하면 답한다.
9. 근거가 부족하면 no-answer로 막는다.

즉 챗봇은 아무 말이나 하면 안 되고:

`근거가 있어야 답하고, 없으면 답하지 말아야` 한다.

---

## 22. 지금 왜 품질이 자꾸 꼬였나

원인은 크게 세 가지다.

### 1. final-only gate가 완전히 잠기지 않았다

최종 파이프라인 문서만 보여야 하는데
일부 surface나 retrieval 조립에서 옛 기준선/중복 family가 섞였다.

### 2. reader-grade contract가 늦게 잠겼다

문단, 코드, 그림 크기 정책이 제품 계약보다 구현 기본값에 더 묶여 있었다.

### 3. trace와 보고가 너무 기술자 중심이었다

무슨 단계가 돌아가는지는 있었지만

- 왜 이 문서가 선택됐는지
- 왜 이 문서는 탈락했는지
- 어느 기술이 어느 역할인지

사용자가 이해할 수 있게 풀어주지 못했다.

---

## 23. 지금 기준으로 정확히 남겨야 하는 것

최종 파이프라인 기준으로 남겨야 하는 건 아래다.

1. active runtime books
2. relation assets
3. figure / diagram assets
4. retrieval index
5. active manifest
6. user overlay
7. runtime evidence

반대로 기준선 밖으로 내릴 후보는 아래다.

1. 예전 md 중심 산출물
2. active가 아닌 processed residue
3. legacy compatibility path
4. 표면에 다시 섞이는 old reference

---

## 24. 진짜 핵심 요약

이 제품은 아래 2문장으로 이해하면 된다.

### 문서 파이프라인

`원문을 가져와서 -> 읽기 좋은 위키 문서로 다시 만든다`

### 챗봇 파이프라인

`그 위키 문서 조각 중에서 -> 질문에 맞는 근거만 찾아 답한다`

그리고 이 두 파이프라인이 제대로 되려면 꼭 지켜야 하는 원칙은 하나다.

`최종 파이프라인 산출물만 active surface에 올라와야 한다`

이게 안 지켜지면:

- 중복 문서
- 엉뚱한 추천질문
- 이상한 관련 문서
- 품질 평가 왜곡

이 반복된다.

---

## 25. 마지막으로 아주 짧게 다시 말하면

- OCR = 이미지 속 글자를 읽는 기술
- BM25 = 단어 비슷한 문서 찾기
- vector = 의미 비슷한 문서 찾기
- GraphDB = 관련된 주변 문서 연결
- reranker = 최종 우선순위 다시 정하기
- viewer HTML = 사용자가 실제로 보는 위키 화면
- runtime markdown/book = 시스템 내부 저장 형식

끝.
